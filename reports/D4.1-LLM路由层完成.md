## 2. 4 个核心模块设计

### 2.1 capability.py — 数据驱动的能力注册表

参考 claw-code `docs/MODEL_COMPATIBILITY.md` 的核心原则：

- **命名约定**：provider/model 格式（如 `deepseek/deepseek-chat`，对应 claw-code `--model "openai/gpt-4.1-mini"`）
- **能力字段**：context_window / supports_chinese / supports_vision / is_reasoning / is_local / priority
- **能力检测函数**：基于 model name 的检测（如 `is_reasoning_model` 识别 o1/o3/o4/grok-3-mini/qwen-qwq/qwen3-thinking）
- **降级默认**：未知模型返回降级 capability（priority=100），避免 None 阻塞 router

**9 个注册模型**（v2 钩子驱动）：

| 类别 | 模型 | priority | 用途 |
|------|------|----------|------|
| 国内默认 | deepseek/deepseek-chat | **10** | 日常路由（性价比最优）|
| 国内推理 | deepseek/deepseek-reasoner | 20 | 智能推理（强制 temperature=1.0）|
| 国内冲榜 | minimax/MiniMax-M3 | 15 | 中文质量（v2 钩子）|
| 国内多模态 | qwen/qwen3-max | 30 | 1M context + 多模态 |
| 国内多模态 | tencent/hunyuan-pro | 40 | v2 钩子：腾讯混元 Hy3 |
| 国内备选 | glm/glm-4-plus | 50 | 智谱 GLM-4-Plus |
| 国外兜底 | anthropic/claude-sonnet-4-6 | 90 | 最终 fallback |
| 国外兜底 | openai/gpt-4.1 | 95 | 1M context 兜底 |
| 本地 | ollama/llama3.2 | 70 | 离线/隐私场景 |

### 2.2 providers.py — Provider 抽象层

参考 claw-code `docs/local-openai-compatible-providers.md`：

- **OpenAI-compatible 统一协议**：国内 + 国外 + 本地都接受 `/v1/chat/completions` + Bearer token
- **环境变量命名**：`OPENAI_BASE_URL` / `OPENAI_API_KEY` / `OLLAMA_HOST`（Ollama 特殊）
- **抽象基类 LLMProvider**：3 个抽象方法（`chat` / `healthcheck` / `provider_type` + `base_url` 属性）
- **占位实现**：D4.1.0 抛 `NotImplementedError`，D4.1.1 实施 httpx + OpenAI SDK 风格

**Factory 函数 `get_provider(full_id)`**：

- 自动按 provider 路由到 base_url（`_BASE_URLS` 表 + Ollama 特殊）
- 未知 provider → `Provider.UNKNOWN` 占位（不抛异常，让 router 跳过）

### 2.3 fallback.py — Fallback 链

参考 claw-code `src/router/fallback.py` 架构原则 + D3.3.3 教训：

- **主选 → 备选 → 兜底链**：3 档异构
- **失败隔离**：单 provider 失败不阻塞链上其他
- **CircuitBreaker**：3 次失败熔断，30 分钟冷却期后自动重置
- **D3.3.3 教训**："异常范围要窄化" — fallback.py 只定义业务错误，编程错误由调用方透传

**5 任务类型 → 5 异构链**：

| TaskType | primary | secondary | tertiary |
|----------|---------|-----------|----------|
| CLASSIFY | deepseek-chat | qwen3-max | M3 |
| DRAFT | M3 | deepseek-chat | qwen3-max |
| ANALYZE | deepseek-reasoner | M3 | GLM-4-Plus |
| STRUCTURE | qwen3-max | hunyuan-pro | deepseek-chat |
| SUMMARIZE | deepseek-chat | M3 | claude-sonnet-4-6 |

**异构性保证**（测试 `test_chain_tiers_distinct` 验证）：3 档必须全异构。

### 2.4 router.py — 5 步决策法主入口

```
1. 接收请求 (task_type + messages + temperature + max_tokens)
2. 查 fallback 链（按 task_type）
3. 遍历 primary → secondary → tertiary:
     a. 熔断检查（is_open? skip）
     b. capability 检查（None? skip）
     c. reasoning 模型强制 temperature=1.0
     d. 调用 provider.chat()
     e. 成功: 记录统计 + 返回
     f. 失败: record_failure + 继续链上
4. 全链失败 → RuntimeError（包含 task_type + last_error）
5. 统计: primary_attempts/successes + fallback_attempts/successes + failures + latency
```

**关键不变量**：

- 主选优先，备选/兜底只在主选失败时启用
- 熔断的 provider 跳过（避免雪崩）
- Reasoning 模型强制 temperature=1.0
- 全链失败抛 `RuntimeError`（让上层 catch 处理）

**单例**：`get_router()` 进程级单例（测试时可显式 `new LLMRouter()` 隔离状态）。

### 3 条架构原则（参考 claw-code 落地）

1. **Capability-Aware**（来源：MODEL_COMPATIBILITY.md）—— router 不硬编码"哪个模型用什么 temperature"，而是查 `cap.is_reasoning` 决定，**新加推理模型只改 registry 不改 router**。
2. **Provider-Agnostic 抽象**（来源：local-openai-compatible-providers.md）—— 6 国内 + 2 国外 + 1 本地共 9 个 provider 全部走同一 `LLMProvider` 接口，**调用方不关心具体 provider 实现**。
3. **Failure-Stop with Circuit Breaker**（来源：claw-code fallback.py 模式 + D3.3.3 教训）—— 3 次失败熔断 + 30 分钟冷却，**避免雪崩**；`router` 只 catch 业务错误不 catch 编程错误（D3.3.3 教训"异常范围要窄化"）。

### 4 个使用警告遵守情况

1. ✅ **不照搬代码**：claw-code `openai_compat.rs` 是 Rust，本项目 Python；只学"模型名 → 能力"映射思想，不抄 Rust trait。

3. ✅ **Python src/ 可直接参考**：`parity_audit.py` 的 `@dataclass(frozen=True)` + 字段驱动设计被 capability.py 借鉴。
