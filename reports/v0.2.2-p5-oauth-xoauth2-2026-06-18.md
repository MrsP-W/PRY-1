### 2.2 RFC 7628 §3.1 SASL 鉴权字符串生成

```python
# 模板(SASL)
XOAUTH2_SASL_TEMPLATE = "user={email}{delim}auth=Bearer {token}{delim}{delim}"
# delim = "\x01" (RFC 7628 §3.1 单元分隔符)

# 拼接 → base64 编码
plain = "user=user@example.com\x01auth=Bearer ya29.xx\x01\x01"
raw = base64.b64encode(plain.encode("utf-8")).decode("ascii")
# → SMTP 鉴权命令
# AUTH XOAUTH2 <base64-encoded-raw>
```

### 2.3 RFC 7628 §3.2 失败响应解析

```python
# 服务器返回 base64-encoded JSON
{"status": "401", "schemes": "bearer", "scope": "https://mail.google.com/"}

# 解析 → XOAUTH2Failure 数据类
failure.status == "401"
failure.schemes == ("bearer",)
failure.scope == "https://mail.google.com/"
failure.is_retryable() == False  # 401 不可重试
```

## 3. 5 段关键设计决策

### 决策 1:XOAUTH2 顶层模块(不嵌入 connectors/smtp.py)

- **原因**:`connectors/smtp.py` 有 14+ import 站点(tests/scripts/policy/e2e),重构为子目录风险高
- **代价**:`connectors/xoauth2.py` 与 `connectors/smtp.py` 同级,而非 launch doc 设想的 `connectors/smtp/xoauth2.py`
- **替代方案**:重构 `smtp.py` → `smtp/__init__.py` + `smtp/xoauth2.py` → 拒(破 14+ import 链)
- **理由充分性**:XOAUTH2 helper 完全独立,无需共享 smtp.py 内部实现,顶层 placement 0 功能影响

### 决策 2:复用 commit 2/3 OAuth2Provider Protocol(不重写)

- **原因**:MicrosoftOAuth2 / GoogleOAuth2 已实现 3 方法(get_auth_url / exchange_code / refresh_token)
- **代价**:XOAUTH2Authenticator 依赖 commit 2/3 抽象层
- **替代方案**:XOAUTH2Authenticator 内部直接调 msal/google_auth → 拒(违反单一职责,与 commit 2/3 抽象层重复)

### 决策 3:4 重防误发(env 门 + factory + 不真发 + email)

- **原因**:D5.6.5 真实 1 封 SMTP 实测教训,防止测试误连真实 outlook/gmail SMTP
- **代价**:XOAUTH2_REAL_NETWORK env 需用户显式设置才能走真实网络
- **替代方案**:仅在 build_auth_string 中严判 → 拒(无法阻止生产环境错误配置)
- **业务价值**:任何 commit 4 测试都可安全跑(默认真实网络关闭)

### 决策 4:数据类 `__post_init__` 双层防御

- **原因**:D4.7.3 v1.0.6 范本 — 数据类 + 工厂层都需要严判,防止直接构造异常数据
- **代价**:XOAUTH2AuthString / XOAUTH2Failure 都有 `__post_init__` 校验
- **替代方案**:仅工厂层严判 → 拒(下游用户直接构造数据类时绕过)

### 决策 5:user_email 作为显式参数(不从 token 提取)

- **原因**:OAuth 2.0 token 不含 user email,Microsoft Graph / Google userinfo 需单独调用
- **代价**:`build_auth_string_via_oauth2_provider` 需 user_email 显式传入
- **替代方案**:在 `exchange_code` 内部自动调用 Graph / userinfo 拿 email → 拒(职责膨胀,违反 commit 2/3 范本)
- **业务价值**:OAuth 2.0 鉴权与 user email 解耦,后续优化可不破坏 helper

## 6. 6 条关键教训沉淀

### 6.1 范本 1:1 复用 commit 2/3 OAuth2Provider Protocol

XOAUTH2 helper 通过 `OAuth2Provider` Protocol 注入 commit 2/3 实现,完全 0 重复代码。建议:**任何后续 OAuth 2.0 业务层(邮件 / 日历 / 通讯录)都沿此范本**,helper 只负责协议层(鉴权字符串),Provider 负责 OAuth 2.0 业务层。

### 6.2 RFC 7628 SASL 是现代标准,JSON 是 legacy

XOAUTH2 默认 SASL 格式(RFC 7628 §3.1),JSON 格式仅作 legacy 兼容。建议:**生产用 SASL**(outlook/gmail 均支持),JSON 仅在老服务器(罕见)用。

### 6.3 OAuth 2.0 token 不含 user email,需外部传入

`build_auth_string_via_oauth2_provider(config, code, user_email)` 显式接收 user_email,不在内部调用 Graph / userinfo。建议:**OAuth 2.0 helper 不应隐式获取 user email**,解耦鉴权与身份查询职责。

### 6.4 4 重防误发是测试安全的基础

`XOAUTH2_REAL_NETWORK=1` 显式 opt-in + `oauth2_provider_factory` mock 注入 + helper 不调 smtplib + email 严判,**4 层叠加确保测试环境无法误连真实 outlook/gmail SMTP**。建议:**任何外部世界集成模块都应沿此范本**。

### 6.5 数据类 `__post_init__` 双层防御(沿 D4.7.3 v1.0.6)

`XOAUTH2AuthString` / `XOAUTH2Failure` 都用 `__post_init__` 严判,防止直接构造异常数据。建议:**所有 frozen 数据类都沿此范本**,即使有工厂层严判,数据类自身也必须自防御。

### 6.6 except 范围窄化(沿 D3.3.3)

`except OAuth2Error: raise` 透传 + `except Exception` 收窄为 XOAUTH2ProviderError,**避免静默吞 OAuth2Error**。建议:**任何业务层 helper 都应沿此范本**,内层异常不静默吞。
