## 2. 核心实现（IMAPSync）

### 2.1 同步流程（[sync.py:80-167](../src/my_ai_employee/core/sync.py#L80-L167)）

```python
async def run_once(self, since=None) -> SyncResult:
    # 1) 读 SyncState.last_uid（首次 0）
    # 2) safe_fetch(since) 拉取（D2 应急版范本：失败隔离 + 熔断）
    # 3) 过滤 uid <= last_uid（旧邮件）
    # 4) 100/批 commit ORM 入库（每批独立 try/except — 单批失败不阻塞）
    # 5) 更新 SyncState（upsert：首次 insert / 后续 update）
    # 6) 返回 SyncResult（fetched/inserted/skipped/failed/new_last_uid/duration）
```

### 2.2 关键设计决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | **100/批 commit ORM** | SQLite 长事务锁风险（>500 行易锁）+ ORM 单 commit 100 行是实测最佳点（250 行 → 3 个 commit 用 ~50ms 端到端）|
| 2 | **失败隔离：单批 `SQLAlchemyError` 不阻塞** | run_once 顶层 `for batch` + 每批独立 try/except — 整批失败计 `failed=N`，下一批继续（**单封失败也要整批跳过**，D3 阶段简化）|
| 3 | **`received_at` 缺失 → fallback `fetched_at`** | D3.1.1 决策（D3 阶段 IMAP raw 不一定带 received_at）|
| 4 | **SyncState upsert（首次 insert / 后续 update）** | D2 应急版范本 6 步：read → 失败隔离 → 拉取 → 入库 → 更新 SyncState → 返回结果 |
| 5 | **关系双轨：本模块只写 `Email.labels` JSON 字段** | D3 阶段快速过滤；`EmailLabel` 关系表由 D4+ LLM 分类时回填 |
| 6 | **`sync.close()` 不 dispose engine** | **D3.2.2 教训**：SA engine 默认 SingletonThreadPool，**复用 db 的 SQLCipher conn**；`engine.dispose()` 会把 db 的 conn 也关掉，触发 "Cannot operate on a closed database" |
| 7 | **scripts/ 不进 mypy 严格区** | CLI/spike 脚本是工具代码，不是业务代码；D3 阶段 `pyproject.toml [tool.mypy] exclude = ["scripts/"]` |

### 2.3 字段映射（_raw_to_email）

```python
def _raw_to_email(self, source, raw, now_ms) -> Email:
    received_at = raw.get("received_at")
    if received_at is None:
        received_at = now_ms  # D3.1.1 决策
    elif isinstance(received_at, datetime):
        received_at = int(received_at.timestamp() * 1000)

    return Email(
        source=source, uid=raw.get("uid", 0),
        message_id=raw.get("message_id"),
        subject=raw.get("subject", ""),
        sender=raw.get("sender", ""),
        recipients=raw.get("recipients", []),  # JSONList 自动转换
        received_at=received_at, raw_size=raw.get("raw_size", 0),
        body_text="", body_html="",  # D4 LLM 分类时回填
        fetched_at=now_ms,
        labels=raw.get("labels", []),  # JSONList 自动转换
    )
```

### 🚨 问题 3 详解：D3.3.1 修复为何实际不生效

**D3.3.1 的修复代码**：
```python
from sqlalchemy.exc import IntegrityError, SQLAlchemyError
...
except IntegrityError:  # ← 这一行不工作
    session.rollback()
    max_uid = 0  # ← 这行从未被执行
    ...
```

**实际错误类型**：当 SQLCipher 触发 UNIQUE 冲突时，session.commit() 抛出的是：
```
sqlcipher3.dbapi2.IntegrityError: UNIQUE constraint failed: emails.source, emails.uid
```

**根因**：SQLCipher (`sqlcipher3`) 不是 SQLAlchemy 官方识别的 dialect，**不包装 DBAPI 异常**。所以 `sqlalchemy.exc.IntegrityError` 永远收不到 — D3.3.1 的 `max_uid=0` 修复逻辑**从未被执行**！

**后果**：UNIQUE 冲突时，IntegrityError 逃逸到 `run_once` 外层 `except SQLAlchemyError` 分支，被错认为"整批失败"（`failed=100` 而非 `skipped=100`），且 `max_uid` 默认是 0（init 值），`new_last_uid = max(last_uid, 0) = last_uid`，**last_uid 确实没被推进**。但**整批邮件的 inserted 计数错误（应为 skipped 100）**，且 D3.3.1 注释里写的逻辑完全没起作用。

**D3.3.2 修复**：
```python
import sqlcipher3.dbapi2 as _sqlcipher_dbapi
...
except (SQLAlchemyError, _sqlcipher_dbapi.IntegrityError):
    try:
        session.rollback()  # 防 session 已 abort 状态二次失败重抛
    except Exception as rb_err:
        logger.warning(f"rollback 失败（已忽略）: {rb_err!r}")
    max_uid = 0
    skipped = len(batch)
    ...
```

**这个 bug 为何复检才发现**：D3.3.1 没有真实重复 UID 的测试覆盖（仅注释里写了"UNIQUE 冲突 → skipped 计数"作为目标）。用户复检建议"给 sync.py IntegrityError 分支加真实重复 UID 测试"，**正是这个测试触发了 bug 暴露**。

### 问题 1 详解：异常范围为何要收窄

**D3.3.2 的修复代码**（D3.3.1 隐藏 bug 暴露后修的）：

```python
except (SQLAlchemyError, _sqlcipher_dbapi.IntegrityError):
    # 整批视作 skipped
    max_uid = 0
    skipped = len(batch)
    ...
```

**问题**：`SQLAlchemyError` 是 SQLAlchemy 所有异常的基类，包括：

- `IntegrityError` — UNIQUE 冲突（**应** skipped）✅
- `OperationalError` — DB 锁 / 文件 IO 错（**应** failed）❌
- `InterfaceError` — DB API 层错（**应** failed）❌
- `DataError` — 数据类型错（**应** failed）❌
- `InternalError` — DB 内部错（**应** failed）❌
- 其他...

**后果**：D3.3.2 的代码会把**所有** SQLAlchemyError 都当 UNIQUE 冲突，**掩盖真实问题**。例如生产环境 DB 锁 5s 后报 OperationalError，本应触发 `failed=N` + `consecutive_failures++` + 熔断检查，但被静默算成 `skipped=N`，运维看不到任何告警。

**D3.3.3 修复**：

```python
except (IntegrityError, _sqlcipher_dbapi.IntegrityError):
    # 真正只处理 UNIQUE 冲突；其他 SQLAlchemyError 抛给外层 run_once 计入 failed
    ...
```

**run_once 外层已有 `except SQLAlchemyError` 兜底**（L134-138）：

```python
for batch_start in range(0, len(new_emails), self._batch_size):
    batch = new_emails[batch_start : batch_start + self._batch_size]
    try:
        b_inserted, b_skipped, b_max_uid = self._commit_batch(source, now_ms, batch)
        ...
    except SQLAlchemyError as e:
        # 整批失败（如 DB 锁）— 不阻塞下一批
        failed += len(batch)
        logger.error(f"批次入库失败（{len(batch)} 封）: {e!r}")
        continue
```

**设计回归**：D3.3 v1.0 设计文档 §2.2 决策 2 写"单批 `SQLAlchemyError` 失败隔离" — 这是 run_once 外层的设计，**不是** `_commit_batch` 内层的设计。D3.3.2 内层把 `SQLAlchemyError` 也接住，**破坏了这个分层**。D3.3.3 修复后回到设计原意。

### D3.3.3 教训（跨 D-step 复用）

1. **异常范围要窄化到"真要处理的类型"**：D3.3.2 的 `(SQLAlchemyError, _sqlcipher_dbapi.IntegrityError)` 是从 D3.3.1 的 `IntegrityError` 过度扩展而来 — 想"接住 SQLCipher 抛的 IntegrityError" 时连同所有 SQLAlchemyError 都接住了。**正确做法**是用 `except (X, Y)` 显式列出要处理的类型，**不要**用基类 catch-all 兜底。
2. **分层 except 各司其职**：`_commit_batch`（内层）只接"业务上等同于 skipped 的错误"（UNIQUE 冲突）；`run_once`（外层）接"整批失败类错误"（DB 锁、连接断开等）。内层不要替外层做决定。
3. **测试要覆盖"非目标路径"**：D3.3.2 的 OperationalError 路径**没有测试**，所以隐藏问题没被发现。D3.3.3 加的 `test_sync_operational_error_propagates_to_failed_not_skipped` 验证"OperationalError **不**被 _commit_batch 接住"，是典型的"非目标路径"覆盖。
