## 4. 设计决策(3 关键)

### 4.1 NoteConfirmService 沿 ExpenseService Protocol 范本

- 3 方法契约(对比 ExpenseService 7 方法): 简化版,只覆盖 L2 1-click 确认业务
- Stub 单例(`get_default_stub()`): 沿 D5.6.4 工厂范本
- Impl(NoteStore 接入): 沿 D10 替换范本,无新 ORM 依赖
- 严判 type 严格(沿 D4.7.3 公共 helper 范本): bool 拒收 / 必填严判

### 4.2 1-click 动作 = NoteStore.mark_archived(apple_note_id)

- 状态机: STRUCTURED → ARCHIVED 终态(沿 [[v0.2.1-candidates-2026-06-17]] §5.2)
- 业务语义: 用户主动决定"这条 L2 候选已合并/独立保留,不再需要 UI 提示"
- 不收容异常: 用户主动操作,必须看到 ValueError(状态机不合法等)

### 4.3 `_update_menu_badge` helper 同时支持 list + rumps.Menu 2 形态

- 形态 1: `list[str]` (test fake_rumps 环境) — `menu[idx] = new_title`
- 形态 2: `rumps.Menu` (真实 NSApp 环境) — `menu.items()` 返回 (str, MenuItem),改 `menu_item.title`
- 关键发现: rumps.Menu 内部是 OrderedDict[str, MenuItem],iter 出来是 str keys;改 MenuItem.title 不会改 keys
- 范本 `_refresh_anomaly_count` 用了 `getattr(item, "title", None)`,在 rumps.Menu 形态下**永远找不到**(str 无 .title 属性)— 本次新写的 helper 正确处理

## 8. 关键教训(5 条)

- **rumps.Menu 内部是 OrderedDict[str, MenuItem]** — iter 出来是 str keys,改 MenuItem.title 不会改 keys;真实 NSApp + test 需用 `menu.items()` 拿 (str, MenuItem) tuple 对
- **范本 `_refresh_anomaly_count` 用 `getattr(item, "title", None)` 永远找不到** — 范本从未被测试覆盖,bug 隐藏;新写的 `_update_menu_badge` helper 正确处理 2 形态
- **fake_rumps fixture 未 monkeypatch `_RumpsAppBase`** — 跨测试文件时真实 rumps 接管 menu 形态,业务代码必须防御性支持 list + rumps.Menu 2 形态
- **mypy Protocol func-returns-value 误报** — `confirm_note() is None` 在 Stub 实现上需 `# type: ignore[func-returns-value]`(Protocol 推断 + Stub return None 的特殊场景)
- **D8.3 异常收容范本应用** — `get_pending_confirm_count` 失败 → 0,`list_pending_confirm` 失败 → [];`confirm_note` 不收容(用户主动操作,必须看到错误)
