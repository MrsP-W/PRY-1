### 2.2 不变量 2: status（7 个状态，5 个负向）

class EventStatus(enum.StrEnum):

### 2.4 不变量 4: fingerprint 去重（跨时间稳定）

```python
def compute_fingerprint(
    event: str | EventType,
    status: str | EventStatus,
    source: str,
    subject_id: str | None,
    metadata: dict[str, Any],
) -> str:
    """SHA-256 派生键(物理去重, g004 不变量 4)"""
    runtime_fields = {"timestamp_ms", "seq", "fingerprint"}
    identity_meta = {k: v for k, v in metadata.items() if k not in runtime_fields}
    payload = {
        "event": event_value,
        "status": status_value,
        "source": source,
        "subject_id": subject_id,
        "metadata": identity_meta,
    }
    canonical = json.dumps(payload, sort_keys=True, ensure_ascii=False)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()
```

**关键设计**：

- **排除运行时字段**（`timestamp_ms` / `seq` / `fingerprint` 自身）→ 同业务事件多次重试 fingerprint 稳定
- **`sort_keys=True` canonical JSON** → key 顺序不影响哈希
- **包含 6 必含字段中的 3 个身份字段**（`ownership` / `provenance` / `session_id`）+ extra payload

测试 `test_dedupe_across_different_time_stays_same` 验证：调用 `build_event_metadata(seq=1, timestamp_ms=1000)` 和 `build_event_metadata(seq=2, timestamp_ms=2000)`，算出的 fingerprint 相同。

### 2.5 UNIQUE 约束 + dedupe

```sql
UNIQUE(fingerprint)
```

- **单字段 UNIQUE(fingerprint) 全局唯一**（D4.3.1 复检 P1 修复: 旧 4 字段联合 UNIQUE 在 `subject_id=NULL` 时被 SQLite 视为不同行, 破坏 dedupe; 改单字段全局唯一后, `subject_id=NULL` + 同 fingerprint 第二次插入必被拒; D4.3.2 增 0003 迁移让已迁移到旧 0002 的库也能升级到此约束）
- **fallback 跨源安全**（deepseek 失败 → openai 重试）：`compute_fingerprint` 入参含 `source`, 不同 source → 不同 fingerprint → 各自 1 条, 不被误判为重复
- **SQLite UNIQUE 不支持函数表达式**（D3.2.3 教训）→ 必须把 `fingerprint` 提为列上索引
- 同一 fingerprint 重复写入**直接去重**（`on_conflict="ignore"` 默认）或**抛异常**（`on_conflict="raise"`）

`store.insert()` 用窄化 except 模式（D3.3.3 教训）：

```python
try:
    session.add(event_obj)
    session.flush()
except IntegrityError as err:
    session.rollback()
    if on_conflict == "ignore":
        return self.get_by_fingerprint(fingerprint)  # 拿到原 id
    raise EventFingerprintConflictError(...) from err  # 窄化 + 保留 stack trace
```

**注意**：`except IntegrityError`（不是 `SQLAlchemyError`），**OperationalError / DataError 等非 UNIQUE 错误不掩盖**。

## 3. events 表 DDL

```sql
CREATE TABLE IF NOT EXISTS events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event TEXT NOT NULL,
    status TEXT NOT NULL,
    source TEXT NOT NULL DEFAULT '',
    subject_id TEXT,
    fingerprint TEXT NOT NULL DEFAULT '',
    event_metadata TEXT NOT NULL DEFAULT '{}',  -- 列名 event_metadata 避开 SA 保留属性
    created_at INTEGER NOT NULL
);

-- 1 UNIQUE + 6 索引
CREATE UNIQUE INDEX uq_events_event_source_subject_fingerprint
    ON events(event, source, subject_id, fingerprint);
CREATE INDEX idx_events_created_at ON events(created_at DESC);
CREATE INDEX idx_events_event ON events(event);
CREATE INDEX idx_events_status ON events(status);
CREATE INDEX idx_events_source ON events(source);
CREATE INDEX idx_events_subject_id ON events(subject_id);
CREATE INDEX idx_events_fingerprint ON events(fingerprint);
```

**关键设计**：

- **列名 `event_metadata`** 而非 `metadata`（SQLAlchemy Declarative 保留属性）— 业务层 / contract / fingerprint 内部仍用 `metadata` 命名（g004 风格，业务友好）
- **`created_at DESC` 索引** — 热路径"倒序拉最近事件"不需要排序
- **6 索引** — 按 `event` / `status` / `source` / `subject_id` / `fingerprint` 5 维度查询都是 O(log n)

## 4. EventStore API

```python
class EventStore:
    def __init__(self, session_factory: sessionmaker) -> None: ...
    def insert(
        self,
        event: str | EventType,
        status: str | EventStatus,
        source: str = "",
        subject_id: str | None = None,
        metadata: dict[str, Any] | None = None,
        on_conflict: Literal["ignore", "raise"] = "ignore",
    ) -> Event: ...
    def get_by_id(self, event_id: int) -> Event | None: ...
    def get_by_fingerprint(self, fingerprint: str) -> Event | None: ...
    def by_session(
        self, session_id: str, status: str | None = None, limit: int = 100
    ) -> list[Event]: ...
    def by_subject(
        self, subject_id: str, limit: int = 100
    ) -> list[Event]: ...
    def by_event_type(
        self, event: str | EventType, limit: int = 100
    ) -> list[Event]: ...
    def by_status(
        self, status: str | EventStatus, limit: int = 100
    ) -> list[Event]: ...
    def count(self) -> int: ...
```

**4 类查询覆盖**：

- **正向查询**（succeeded / started）：`by_event_type(LLM_CALL_SUCCEEDED)`
- **负向查询**（failed / degraded / skipped / blocked / cancelled）：`by_status(FAILED)` 拿失败清单
- **按 session 聚合**（端到端追踪）：`by_session("sess-123")`
- **按 subject 聚合**（按实体追踪）：`by_subject("email-456")` 拿这封邮件的所有事件

**`on_conflict` 双模式**：

- `"ignore"`（默认）— dedupe 命中是正常业务（重试、网络抖动重发），返回原 id 不抛错
- `"raise"` — 严格模式（如审计场景），UNIQUE 冲突显式抛 `EventFingerprintConflictError`

## 5. JSONDict TypeDecorator

```python
class JSONDict(TypeDecorator):
    """dict ↔ JSON 文本 TypeDecorator (D4.3 新增, 与 D3.2 JSONList 同模式)."""
    impl = Text
    cache_ok = True

    def process_bind_param(self, value, dialect):
        """ORM → DB: dict 序列化为 JSON 文本. None → None."""
        if value is None:
            return None
        return json.dumps(value, ensure_ascii=False)

    def process_result_value(self, value, dialect):
        """DB → ORM: JSON 文本 → dict. 空字符串/None 一律视作 {}."""
        if not value:
            return {}
        return json.loads(value)
```

**为什么单独类**（不沿用 D3.2 JSONList）：

- dict 与 list 业务语义不同（事件 payload vs 邮件收件人列表）
- 独立类名让 import 更清晰（`JSONDict` 一看就知道是 dict 类型）
- TypeDecorator 内部逻辑可独立演进（dict 序列化/反序列化与 list 不同）

**为什么不用 SQLAlchemy 自带 `JSON` 类型**：

- D3.2 决定所有 JSON 字段用 `TEXT` + `dumps/loads`（[memory/d3.2-orm-migrations.md §设计决策](../../../../../Agent%20Assistant/memory/d3.2-orm-migrations.md)）
- 统一存储层（schema.sql 也用 `TEXT DEFAULT '{}'`）
- mypy 友好（`Mapped[dict[str, Any]]` 不需要 `JSON` 的 generic）

## 11. D4.3.1 复检修复（6/8 复检后立即修复）

### 11.1 P1 修复：fingerprint 全局 UNIQUE

**根因**：旧 `UNIQUE(event, source, subject_id, fingerprint)` 4 字段联合约束，在 `subject_id=NULL` 时被 SQLite 视为不同行，破坏 dedupe。复现：

```text
# 1st insert id=1, fingerprint=91925386f6b6a652
# 2nd insert id=2 ← P1 漏洞: 未抛异常 (subject_id=NULL)
# total rows: 2
```

**修复**：`UNIQUE(fingerprint)` 全局唯一键。

**fallback 跨源场景安全性**：`compute_fingerprint()` 入参含 `source`，不同 source（deepseek 失败 + openai 重试）→ 不同 fingerprint → 各自 1 条，不会被误判。新增回归测试 `test_dedupe_fallback_cross_source_allowed` 验证。

**变更点**：

### 11.2 P2 修正：契约口径精确化

> "4 大不变量在 EventStore.insert() 调用栈内保证":
> - ✅ 业务代码走 `EventStore.insert()` → 不变量由 `build_event_metadata` + `assert_event_invariants` + `compute_fingerprint` + `UNIQUE(fingerprint)` 联合保证
