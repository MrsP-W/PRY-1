## 2. 🎯 设计决策（8 大关键点）

### 2.1 TaskPacket 8 字段全 default（向后兼容）

**问题**：g006 §"Task packet schema" 要求 8 必含字段，但 D4.5+ 业务代码可能从旧 JSON 反序列化（如 sync config / 任务历史），缺字段怎么办？

**方案**：8 字段全部 `field(default=...)` + `TaskPacket.from_dict(data)` 走 `data.get("xxx", default)` 模式。

**示例**：

```python
# 旧 JSON 缺字段也能 load
packet = TaskPacket.from_dict({})
assert packet.objective == ""  # 不是 KeyError
```

**验证**：`tests/policy/test_task_packet.py::TestBackwardsCompat::test_from_dict_missing_fields_uses_defaults`

### 2.2 5 类业务异常窄化（D3.3.3 教训应用）

**问题**：D3.3.3 教训：异常范围要窄化到真要处理的类型，不能 `except Exception` 兜底。

**方案**：`PolicyError` 基类 + 5 子类（Contract / Decision / Approval / Heartbeat / Lane），每类对应一类业务错误。

**对照**：

| 异常 | 触发场景 | 透传还是抛 |
|------|---------|----------|
| `PolicyContractError` | TaskPacket 8 字段不变量校验失败 | 抛 |
| `PolicyDecisionError` | context 类型非法 / rule 自身抛错 | 抛 |
| `PolicyApprovalError` | 审批 token 缺失/失效（D4.4 占位） | D4.5+ 真实触发 |
| `PolicyHeartbeatError` | `assert_alive()` 在 TRANSPORT_DEAD 时 | 抛 |
| `PolicyLaneError` | LaneBoard 状态转换非法 | 抛 |
| `ValueError` / `TypeError` | 编程错误（参数类型错 / 时间倒流） | 透传 |

**验证**：`tests/policy/test_exceptions.py` 10 个测试 + 5 子类 `assert issubclass(cls, PolicyError)` 锁定层级。

### 2.3 Heartbeat 优先级：TRANSPORT_DEAD > STALLED > HEALTHY

**问题**：transport 断连（mcp server 离线）vs 心跳超时，哪个优先级更高？

**方案**：`evaluate()` 第一步先检查 `transport_alive`，False 直接返回 TRANSPORT_DEAD，不论 last_seen 多近。

**为什么**：transport 断连 = 必失败，调用方应立即升级/重连；心跳超时 = 可能只是 idle，stuck 但 transport 还活着。

**验证**：`tests/policy/test_heartbeat.py::TestEvaluatePriority::test_transport_dead_overrides_healthy`

### 2.4 LaneBoard 状态转换合法性矩阵

**问题**：3 lanes 间的状态转换是否任意？

**方案**：状态转换矩阵（合法转换 = 显式列出，非法 = 抛 PolicyLaneError）：

```
ACTIVE   → {BLOCKED, FINISHED}     (等审批 / 正常完成)
BLOCKED  → {ACTIVE, FINISHED}      (恢复 / 取消)
FINISHED → ∅                        (终态, 不能回退)
```

**额外约束**：`add()` 不允许直接 FINISHED 状态（终态需走 ACTIVE → FINISHED 路径）；相同状态幂等通过（便于 caller 重复 update）。

**验证**：`tests/policy/test_lane_board.py::TestStateTransitions` 15 个测试覆盖 6 个合法转换 + FINISHED 终态 + 非法转换抛错。

### 2.5 6 决策 priority 排序（priority 越高越先执行）

**问题**：6 个决策可能同时触发，caller 按什么顺序执行？

**方案**：每个 decision 有 `priority: int`（0-100），`PolicyEngine._run_all_rules()` 跑完所有 rule 后 `decisions.sort(key=priority, reverse=True)`。

**当前 priority 分布**：

| Decision | Priority | 触发条件 |
|----------|----------|----------|
| `EscalateRequired` | 100 | policy_eval_failed=True（最高）|
| `ApprovalTokenRequired` | 80 | READ_WRITE/ADMIN + 无 token |
| `RetryAvailable` | 70 | recoverable + attempts<max |
| `RebaseRequired` | 60 | branch_stale=True |
| `StaleCleanupRequired` | 50 | heartbeat 超时 |
| `MergeRequired` | 40 | 全部 acceptance 通过 |

**验证**：`tests/policy/test_policy_engine.py::TestEvaluateEntry::test_decisions_sorted_by_priority_desc`

### 2.6 事件落地复用 D4.3 EventStore

**问题**：D4.4 决策事件需要持久化吗？新建表还是复用 events 表？

**方案**：复用 D4.3 `EventStore.insert()`，新增 2 个 `EventType` 枚举值（`POLICY_DECISION_MADE` / `POLICY_DECISION_DEGRADED`），不引入新表。

**extra payload 字段**（7 个业务字段）：

| 字段 | 类型 | 来源 |
|------|------|------|
| `rule_name` | str | 主决策 rule_name |
| `priority` | int | 主决策 priority |
| `kind` | str | 主决策 kind |
| `explanation` | str | 主决策 explanation |
| `approval_token_id` | str | 主决策 approval_token_id（可能空）|
| `all_decisions` | list[dict] | 全部 6 rule 触发结果（按 priority 排）|
| `context_snapshot` | dict | 评估时 context（便于复现）|

**验证**：`tests/policy/test_policy_engine.py::TestEventStoreIntegration` 7 个测试覆盖 事件落地 + event_type + metadata + fingerprint dedupe + ownership/provenance。

### 2.7 now_ms 时间注入（避免 time.time() 依赖）

**问题**：`Heartbeat.update()` / `LaneBoard._compute_freshness()` / `PolicyEngine._rule_stale_cleanup_required()` 都依赖当前时间，但测试要可控。

**方案**：所有时间相关方法接受 `now_ms: int | None = None` 参数，None 时回退 `int(time.time() * 1000)`。

**验证**：所有 policy 测试都用注入的 now_ms（避免 sleep / time 漂移）。

### 2.8 编程错误透传（D3.3.3 教训）

**问题**：ValueError / TypeError 是编程错误，包装成业务异常会掩盖问题。

**方案**：

- `Heartbeat.__post_init__` 校验 `idle_threshold_ms` 类型 → 抛 `ValueError`（透传）
- `LaneBoard.add()` 校验 `entry_id` 类型 → 抛 `PolicyLaneError`（业务错误）
- `LaneBoard.update()` 校验 `owner` 类型 → 抛 `ValueError`（透传）
- `PolicyEngine._normalize_context()` 校验 context 类型 → 抛 `PolicyDecisionError`（业务错误）

**判断标准**：**值是 user 提供的不可信数据 → 业务异常；值是 caller 必传对的数据 → 编程错误（ValueError 透传）**。

        .build()

        "max_attempts": 3,
