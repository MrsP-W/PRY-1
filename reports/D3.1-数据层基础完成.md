### 2.1 DB 密码自动生成 + Keychain 隔离

| 维度 | 决策 | 理由 |
|------|------|------|
| 密码生成 | `secrets.token_hex(32)`（64 chars）| 密码学安全随机，零记忆负担 |
| 密码存储 | macOS Keychain service=`my-ai-employee.db` account=`data.db` | 与 IMAP 授权码**完全隔离**（不复用 D2.3 IMAP 凭证）|
| DB 路径 | `~/Library/Application Support/my-ai-employee/data.db` | macOS 标准 Application Support 目录 |
| 首次启动 | Keychain 缺密码 → 自动生成 + 写入 | 零配置开箱即用，密码丢失 = 数据无法解密（已记录在 [memory/d2-imap-real-qq-verified.md](file:///Users/wei/.claude/projects/-Users-wei-Documents-DesktopOrganizer-Agent-Assistant/memory/d2-imap-real-qq-verified.md)） |

### 2.2 PRAGMA key 密码设置

```python
# 错误方案 1: conn.execute("PRAGMA key = ?", (password,))
# → "OperationalError: near '?': syntax error"（PRAGMA 不支持参数化占位符）

# 错误方案 2: conn.execute(f"PRAGMA key = '{password}'")
# → 密码含 `'` 时引爆 SQL

# ✅ 正确方案: executescript + 单引号转义
safe_password = password.replace("'", "''")
conn.executescript(f"PRAGMA key = '{safe_password}'; PRAGMA foreign_keys = ON;")
```

**为什么用 executescript**：sqlite `PRAGMA` 不支持 `?` 参数化，必须走 SQL 解析器。`executescript` 内部用 sqlite3 解析器，能吃字符串拼接 + 多语句（顺便把 `PRAGMA foreign_keys = ON` 也合进去）。

### 2.3 行工厂：dict_factory 而非 sqlite3.Row

```python
def _dict_factory(cursor, row):
    return {col[0]: row[idx] for idx, col in enumerate(cursor.description)}

conn.row_factory = _dict_factory
```

**坑**：sqlcipher3 的 `Cursor` **不是** `sqlite3.Cursor` 子类。直接用 `sqlite3.Row` 工厂会抛：

```
TypeError: Row() argument 1 must be sqlite3.Cursor, not sqlcipher3.dbapi2.Cursor
```

**方案**：自定义 `dict_factory`，行为等价（`row["col"]` 可用），且不依赖 cursor 类型。

### 2.4 主动密码校验：PRAGMA quick_check

```python
# SQLCipher 是懒校验的 — PRAGMA key 本身不读加密页
# 错误密码要到第一次 SELECT/INSERT 时才报 "file is not a database"
# 错误位置远离 root cause，用户 debug 痛苦

# 试错的触发方式（不靠谱）:
# - SELECT count(*) FROM sqlite_master   → page 1 header 不加密，错密码也能跑
# - PRAGMA cipher_version                 → 编译时常量，不读加密页
# - PRAGMA integrity_check                → 可靠但读全库，MemoryError 风险

# ✅ PRAGMA quick_check → 轻量 + 100% 触发
try:
    conn.execute("PRAGMA quick_check").fetchone()
except sqlcipher3.DatabaseError as e:
    conn.close()
    raise sqlcipher3.DatabaseError(f"DB 打开失败：密码错误或文件损坏 ({e})") from e
```

**实战经过**：原本想用 `SELECT * FROM emails LIMIT 1` 触发，但**全新空 DB 没有 emails 表** → 抛 "no such table" → 被误判为密码错。这个鸡生蛋问题用 `PRAGMA quick_check` 完美解决（不依赖任何表存在，只读 schema root page）。

## 4. 已知雷区 + 解决方案

| # | 雷区 | 触发 | 解决 |
|---|------|------|------|
| 1 | PRAGMA 不支持 `?` 占位符 | `conn.execute("PRAGMA key = ?", ...)` | `executescript` + 字符串拼接 + 单引号转义 |
| 2 | sqlcipher3 Cursor 不是 sqlite3.Cursor | `conn.row_factory = sqlite3.Row` | 自定义 `_dict_factory(cursor, row) → dict[列名, 值]` |
| 3 | SQLCipher 懒校验 — PRAGMA key 不报密码错 | 错密码 → 用 SELECT 等操作时才报 | `Database.open()` 主动跑 `PRAGMA quick_check` 触发 |
| 4 | SELECT count(*) FROM sqlite_master 不触发校验 | 空库/小 schema | 改用 `PRAGMA quick_check`（不依赖表存在）|
