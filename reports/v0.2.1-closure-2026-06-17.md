## 🎯 v0.2.1 收官 5 关键教训(2026-06-17 沉淀)

- **候选清单 docs-only 整理 = 6 候选锁定 = 9 commits 收口链** — 沿 v0.1.0-closure-preview 范本,candidates 文档先做 3 维度评估(工作量 / 依赖 / 风险),代码实化沿优先级矩阵解封,9 commits 链 1:1 对应 6 候选 + 1 增量 + 1 drift fix。
- **SQLite 自引用 FK 在 alembic 中限制** — `op.create_foreign_key` 自引用同表 FK 在 SQLite 上不支持,即便配 `render_as_batch=True`;**解法**:沿 0013 范本无 FK 软标记,字段纯 Integer,引用一致性由应用层 `select Note.id` 校验(已 select,无新增查询)。**应用场景**:自引用 FK + 多步迁移中需要重建 FK + SQLite + alembic 任何带 FK 的 DDL。
- **B 类决策延后声明 = v0.2.1 收口的合规边界** — 沿 [[b-class-deferral-2026-06-09]] 规则,6 候选中 outlook/gmail SMTP provider 是用户单独决策项,代码实施由用户显式解封,不主动提醒,不预先实施。
- **v0.2.1+ 增量 = 1-click 确认 UI 基础设施 100% 落齐** — `needs_confirm` + `candidate_match_id` 双字段 + `list_by_needs_confirm` 热路径 + `idx_notes_needs_confirm` 索引,UI 层只需 SELECT 列表 + 1-click 按钮,**写入侧已无后续 commit**。
- **v0.2.1 docs-only 收口不封 tag** — 沿 D5.7.2 范本 "v0.1.0 tag 锚 2af775f 不动,docs/fix commit 衍生 hash",v0.2.1 release tag 锚定 W3 真账单 spike 跑通后,避免"docs 收口但代码未跑真数据 = 假完成"。
