### 2.2 工厂注入(沿 commit 2 范本)

```python
def __init__(
    self,
    *,
    google_auth_client_factory: Callable[..., Any] | None = None,
    default_scopes: tuple[str, ...] = GOOGLE_DEFAULT_SCOPES,
) -> None:
    self._google_auth_client_factory = google_auth_client_factory
    self._default_scopes = _validate_default_scopes(default_scopes)
```

**设计原则**:测试用 `google_auth_client_factory` 注入 mock,生产用真实 `google_auth_oauthlib.Flow`(沿 commit 2 `msal_client_factory` 范本)。

### 2.3 函数内 deferred import

```python
if TYPE_CHECKING:
    import google_auth_oauthlib  # noqa: F401

def _build_google_auth_client(self, config: OAuth2Config) -> Any:
    if self._google_auth_client_factory is not None:
        return self._google_auth_client_factory(...)
    import google_auth_oauthlib  # 运行时才 import
    return google_auth_oauthlib.flow.Flow.from_client_config(...)
```

**设计原则**:6/19 commit 3 暂不引入 `google-auth` 依赖,函数内 import。6/22 commit 5 加 dep + uv lock(沿 commit 2 时序)。

### 2.4 except 范围窄化(沿 d3.3.3 范本)

```python
try:
    result = client.fetch_token(...)
except OAuth2Error:
    raise  # 透传(不静默吞)
except Exception as e:
    raise OAuth2TokenExchangeError(
        f"GoogleOAuth2.exchange_code: google_auth_oauthlib.Flow.fetch_token 失败: {e!r}"
    ) from e
```

### 2.5 Google 特色 3 字段(关键差异)

```python
params = {
    "client_id": validated_config.client_id,
    "response_type": "code",
    "redirect_uri": validated_config.redirect_uri,
    "scope": scope_str,
    "state": final_state,
    # Google 必含 3 字段(沿 https://developers.google.com/identity/protocols/oauth2/web-server)
    "access_type": "offline",          # 必返 refresh_token
    "prompt": "consent",               # 强制同意页(避免已授权用户跳 consent 跳过 → refresh_token 不颁)
    "include_granted_scopes": "true",  # 沿 Google 文档(增量授权)
}
```

**业务含义**:`access_type=offline` + `prompt=consent` 是 Google 颁发 refresh_token 的**必要条件**(范本:Google docs)。Microsoft 只需 `offline_access` scope 即可。

## 3. 5 段关键设计决策

### 决策 1:沿 commit 2 范本 1:1 复用 6 严判 helper

- **原因**:D4.7.3 v1.0.6 公共 API 自防御范本已稳定,MicrosoftOAuth2 验证过(12 tests 全过),GoogleOAuth2 1:1 复用避免重新发明
- **代价**:helper 文件略长但测试覆盖完整
- **替代方案**:仅复用抽象层 OAuth2Config → 拒(严判需在 Provider 入口,不在抽象层)

### 决策 2:工厂注入 google_auth_client_factory

- **原因**:6/22 commit 5 才加 dep,测试需 6/18 跑通 → 工厂注入解耦测试与生产
- **代价**:多 1 个构造参数
- **替代方案**:直接在 `__init__` import google_auth_oauthlib → 拒(dep 6/22 才加,过早引入导致 6/18 测试失败)

### 决策 3:Google 特色 3 字段(必含,不可省)

- **原因**:`access_type=offline` + `prompt=consent` 是 Google refresh_token 颁发的硬性要求(范本 Google docs),缺一即丢失长期鉴权能力
- **代价**:URL 字符串变长(~50 字符)
- **替代方案**:不传 access_type → 拒(refresh_token 不颁发,整个 OAuth 鉴权链路不可持续)

### 决策 4:deferred import(函数内 import google_auth_oauthlib)

- **原因**:6/22 commit 5 才加 dep,commit 3 测试不能依赖真实 google_auth_oauthlib(否则 6/18 跑不通)
- **代价**:函数内 import 略慢(但只在 client 构造时调用 1 次)
- **替代方案**:在文件顶部 import → 拒(过早依赖 6/22 才加的 dep)

### 决策 5:except 收窄(仅捕非 OAuth2 异常)

- **原因**:D3.3.3 范本(异常收窄,OperationalError 风格传播),避免吞掉 OAuth2 异常
- **代价**:需多 1 个 except 分支
- **替代方案**:直接 except Exception → 拒(D3.3.3 教训,过宽会掩盖真实生产问题)

## 6. 5 条关键教训沉淀

### 6.1 范本 1:1 复用的边界

commit 2 (MicrosoftOAuth2) 范本稳定后,commit 3 (GoogleOAuth2) 直接 1:1 复用 6 helper + factory + deferred import,几乎零新风险。建议:**OAuth 2.0 Provider 实现统一沿用此范本**,任何新 Provider(Yahoo / iCloud 等)只需替换 endpoint + scope + 工厂即可。

### 6.2 Google 特色字段不可省

`access_type=offline` + `prompt=consent` 是 Google 颁发 refresh_token 的硬性条件。Microsoft 只需 `offline_access` scope 即可,**两端默认 scope 设计完全不同**。建议:OAuth 2.0 Provider 文档化"特色字段"差异。

### 6.3 deferred import 是测试解耦的核心

函数内 `import google_auth_oauthlib`(配合 TYPE_CHECKING)+ 工厂注入,使得 6/19 commit 3 在**未引入 google-auth 依赖**前即可跑通 11 个测试。沿用 commit 2 范本,无新风险。

### 6.4 except 范围窄化(沿 D3.3.3)

OAuth2Error 透传 + Exception 收窄为 OAuth2TokenExchangeError/RefreshError。**过宽会掩盖真实生产问题**(DB 锁 / 网络错 / 鉴权错 等需区分处理)。沿用 D3.3.3 范本,无新风险。

### 6.5 测试覆盖矩阵完整(11 cases / 4 段)

URL 构造(5)+ exchange_code(2)+ refresh_token(2)+ 严判 + Protocol 合规(2)= 11 tests,覆盖边界条件 + 成功路径 + 异常收窄 + Protocol 合规 4 维度。沿用 commit 2 范本,无新风险。
