## 4. 设计决策(3 关键)

### 4.1 L3 模糊匹配 = 商家名归一化 + 日期 ±N day 容错

**算法**(沿 [[v0.2.1-candidates-2026-06-17]] §6 候选 #5 描述):
```
1. 归一化商家名(复用 _normalize_counterparty_value)
   strip + lower + 去模糊符 * + 去所有空白
2. 圈定日期窗口 ±N day(N 默认 1,允许 0-7 范围)
3. 全表扫 transaction_date 范围(D6 现状无 transaction_date 索引)
4. 内存过滤:归一化商家名相等 + 排除自身
5. 排序 + LIMIT(沿 L2 范本 id ASC)
```

**严判优先原则**:
- 宁可漏匹配(false negative),不要误匹配(false positive)
- 用户 1-click 确认前 L3 候选已标红,用户可手动否决
- 模糊匹配误命中会让用户困惑"为什么这两个看起来不相关?" — 严判 > 召回

### 4.2 L3 归一化比 L2 更激进(Notes 场景)

**L2 vs L3 归一化对比**:

| 字段 | L2 归一化 | L3 归一化 |
|------|----------|----------|
| title (Notes) | strip + lower | strip + lower + 去 * + 去所有空白 |
| counterparty (Transactions) | strip + lower + 去 * + 去所有空白 | (同 L2,复用 _normalize_counterparty_value) |

**为什么 L3 更激进**:
- L2 是确定性指纹(SHA-256),归一化必须稳定可复现
- L3 是"容错查询",容许更多差异(用户笔记"星巴克*"脱敏写法)
- L3 复用 `_normalize_counterparty_value` 范本,因为该范本已处理 * 模糊符

### 4.3 L3 fallback 触发:仅 L2 不命中时

**Notes insert 流程**(沿 D6.4 transactions L2 范本):
```
2 段: 派生 normalized_fingerprint
2.5 段: L2 查 fingerprint → 命中设 needs_confirm=1
2.6 段: L2 未命中时 L3 模糊匹配 → 命中设 needs_confirm=1
3 段: INSERT + commit
```

**为什么不替代 L2**:
- L2 命中率高(同商家同日同金额)→ L2 应优先
- L3 容错只在 L2 不命中时触发,防"L3 把 L2 已经命中的候选替换成错的"
- L3 候选集 = L2 候选集的"扩展",不重叠

## 8. 关键教训(5 条)

- **L3 复用已有归一化函数** — `_normalize_counterparty_value` 沿用,不重新发明;严判稳定可复现
- **L3 归一化比 L2 激进**(去 * + 去空白)— L3 是容错查询,与 L2 确定性指纹区分
- **L2 → L3 fallback 模式** — L2 命中优先(命中率高),L3 仅 L2 未命中时触发(防 L3 误替换)
- **严判 type(value) is bool 陷阱** — `isinstance(True, int)==True`,必须用 `type() is bool` 严判
- **空 title L3 静默跳过** — 严判失败容错(空归一化 L3 无可匹配目标,直接返回 [])
- **datetime ISO 字符串不能直接相减** — `_normalize_note_updated_at_date` 返回 str,需 `strptime` 转 date 才能算 timedelta
- **业务层 L2 vs L3 一致性** — 同一个 `candidate_match_id` 字段语义,UI 层 list_by_needs_confirm 一视同仁消费
