## 3. LLM 路由策略修正（与原 PersonalOS 不同）

### 旧方案（PersonalOS 阶段）

| 数据类型 | 主路由 | Fallback |
|----------|--------|----------|
| 邮件 | Claude API | 本地 Ollama |
| 敏感数据 | 本地 Ollama | 跳过 LLM |

### 新方案（我的AI员工，已确认）

| 数据类型 | 主路由 | Fallback |
|----------|--------|----------|
| 所有非敏感 | **minimax M3** | 规则引擎（关键词/正则）|
| 敏感数据 | **跳过 LLM**（标记 `pending`）| 用户手动处理 |

### 理由

1. **统一链路** — 与 Agent Assistant 现有链路一致
2. **减少复杂度** — 不装 Ollama（4-5GB 模型下载 + 性能调优）
3. **质量优先** — minimax M3 中文分类比 qwen2.5:7b 强 1-2 个量级
4. **数据量小** — 个人日均邮件 50-100 封，敏感占比 < 5%
5. **Phase 2 可加** — 如果某类敏感场景频繁，针对性加本地 LLM

### 降级路径（4 级，去掉 Ollama）

1. **L1 主路径** — 全功能（minimax M3）
2. **L2 规则降级** — LLM → 关键词/正则
3. **L3 只读模式** — 写入暂停，只读历史
4. **L4 完全离线** — 仅本地数据，不联网

## 5. 关键工程决策

### 5.1 应急版范本（main.py 降级）

**问题**：D1 阶段 poetry 未装时，`rich` 不可用，`make hello` 会崩。

**方案**：`main.py` 检测 `rich` 是否可用，缺失时降级到纯文本 panel（应急版范本 L3）。

**效果**：
- 有 venv/poetry → 彩色 rich 面板
- 无 venv/poetry → 纯文本 panel（提示"应急版"）
- 两种模式都通过 6 个测试

### 5.2 命令模式（python -m src.main）

**问题**：`python src/main.py` 失败，因为 Python 把 `main.py` 当顶层脚本而非 `src` 包。

**方案**：所有命令用 `python -m src.main`，Makefile 自动用 `.venv/bin/python` 或 `poetry run python`。

### 5.3 Makefile 自动检测 Python

```makefile
PYTHON := $(shell \
    if [ -x .venv/bin/python ]; then \
        echo ".venv/bin/python"; \
    elif command -v poetry >/dev/null 2>&1; then \
        echo "poetry run python"; \
    else \
        echo "python3"; \
    fi)
```

**效果**：
- 有 `.venv/` → 直接用（最快）
- 有 `poetry` → 走 poetry
- 都没有 → 降级到 `python3`（应急）

### 5.4 poetry install 失败的处理

**问题**：Python 3.14 + certifi 冲突 + PEP 668 锁，poetry 装包失败。

**解决路径**：
1. 移除冲突的 `certifi-2026.5.20.dist-info` — 失败
2. 创建 `.venv/` 用 uv 绕过 PEP 668 — 成功
3. 用 `uv pip install` 装核心依赖 — 成功
4. poetry 与 .venv 并存（Makefile 优先用 .venv）

**状态**：
- poetry 2.4.1 已装（brew）
- `.venv/` 已建（uv）
- 核心依赖已装（rich/pytest/pydantic/...）
- 完整 poetry 依赖（imapclient/sqlcipher/...）**留待 D2+ 装**

## 7. 关键发现 & 经验

### 7.1 Python 3.14 兼容性

- **优势**：速度更快、类型系统改进
- **劣势**：第三方包兼容性差（PEP 668 + 库 wheel 缺失）
- **D1 决策**：用 `.venv/` + uv，绕开系统 Python 锁

### 7.2 poetry vs uv

- **poetry 优势**：锁定更严格、生态成熟
- **uv 优势**：速度快、不受 PEP 668 限制
- **D1 决策**：**双轨制**——pyproject.toml 用 poetry 标准格式（用户决策），但实际安装用 uv pip（环境所迫）

### 7.3 路径含中文

- macOS 系统路径含中文（`我的AI员工/`）目前无问题
- `tests/__init__.py` 中 `Path(__file__)` 解析正常
- **风险**：Phase 2 加 subprocess 时需测试中文路径参数传递

### 7.4 rich 降级模式

- `try: from rich import ... except ImportError` 是简单可靠模式
- 降级版**仍能展示项目基本信息**（不丢失信息）
- 应急版范本 L3（降级到本地）—— LLM 调用未来也会用同样模式
