## 1. 4 类业务异常（D3.3.3 教训落地）

参考 D3.3.3："异常范围要窄化到真要处理的类型"——`router.route()` 只对**业务异常**触发 fallback，**编程错误**（参数错）必须直接抛。

### 1.1 异常分层

```
httpx.TimeoutException
  → LLMTimeoutError        # 超时 → 触发 fallback
httpx.ConnectError
httpx.RequestError(其他)
  → LLMConnectionError     # 网络错 → 触发 fallback
HTTP 4xx/5xx
  → LLMAPIError(status, body)  # API 错 → 触发 fallback
非 JSON / 缺字段 / 字段类型错
  → LLMResponseError       # 响应解析错 → 触发 fallback

ValueError / TypeError    # 编程错误 → 透传, 不包装
```

### 1.2 为什么不 catch-all `Exception`

**D3.3.3 教训**（[memory/D3.3-sync-script.md](../../../../../Agent%20Assistant/memory/D3.3-sync-script.md)）：

> `(SQLAlchemyError, _sqlcipher_dbapi.IntegrityError)` 过宽 — `SQLAlchemyError` 是基类，会把 OperationalError / DB 锁 / InterfaceError / DataError 等非 UNIQUE 错误**全误算为 skipped**，**掩盖真实生产问题**。

D4.1.1 应用同样的原则：

- `LLMError` 是 4 类业务异常的基类，**不是 `Exception` 兜底**
- `ValueError` / `TypeError`（编程错误）**透传**，让 router 上层 / 调用方知道是"代码 bug"而非"网络问题"
- 测试 `TestChatProgrammingError` 验证编程错误不被包装为 `LLMError`

### 1.3 LLMAPIError 的 body 截断

```python
def __init__(self, message: str, status_code: int, body: str) -> None:
    self.body = body[:500]  # 截断 500 字符, 防止巨型响应爆日志
```

- 真实场景：某些 provider 错误响应含完整 stack trace（10KB+），不截断会污染日志
- 测试 `test_api_error_truncates_huge_body` 验证：`"x" * 5000` → `len(body) == 500`

## 2. 实施细节

### 2.1 chat() 流程

```python
def chat(self, request: LLMRequest) -> LLMResponse:
    # 1. 构造请求体（从 full_id 拆 model_id）
    if "/" not in request.model_full_id:
        raise ValueError(f"model_full_id 格式错误: ...")  # 编程错误透传
    model_id = request.model_full_id.split("/", 1)[1]
    url = f"{self._base_url}/chat/completions"
    payload = {
        "model": model_id,
        "messages": list(request.messages),
        "temperature": request.temperature,
        "max_tokens": request.max_tokens,
    }
    headers = {
        "Authorization": f"Bearer {self._api_key}",
        "Content-Type": "application/json",
    }

    # 2. 同步 httpx（每次新建短连接，避免连接泄漏）
    start = time.monotonic()
    try:
        with httpx.Client(timeout=self._timeout) as client:
            response = client.post(url, json=payload, headers=headers)
    except httpx.TimeoutException as e:
        raise LLMTimeoutError(...) from e
    except httpx.ConnectError as e:
        raise LLMConnectionError(...) from e
    except httpx.RequestError as e:
        raise LLMConnectionError(...) from e

    latency_ms = int((time.monotonic() - start) * 1000)

    # 3. 检查 HTTP 状态码
    if response.status_code >= 400:
        raise LLMAPIError(...)

    # 4. 解析响应（OpenAI 风格）
    try:
        data = response.json()
        content = data["choices"][0]["message"]["content"]
        usage = data.get("usage", {})
        input_tokens = int(usage.get("prompt_tokens", 0))
        output_tokens = int(usage.get("completion_tokens", 0))
    except (KeyError, IndexError, TypeError, ValueError) as e:
        raise LLMResponseError(...) from e

    return LLMResponse(content=..., model_full_id=..., input_tokens=..., output_tokens=..., latency_ms=latency_ms)
```

### 2.2 API Key 解析策略

```python
_API_KEY_ENV: dict[Provider, str] = {
    Provider.DEEPSEEK: "DEEPSEEK_API_KEY",
    Provider.QWEN: "DASHSCOPE_API_KEY",
    Provider.GLM: "GLM_API_KEY",
    Provider.MINIMAX: "MINIMAX_API_KEY",
    Provider.TENCENT: "TENCENT_API_KEY",
    Provider.OPENAI: "OPENAI_API_KEY",
    Provider.ANTHROPIC: "ANTHROPIC_API_KEY",
}

def _resolve_api_key(provider: Provider, override: str | None) -> str:
    if override:
        return override
    env_name = _API_KEY_ENV.get(provider, "OPENAI_API_KEY")
    return os.environ.get(env_name, "") or os.environ.get("OPENAI_API_KEY", "")
```

**优先级**：`override` 参数 > 专用 Key（DEEPSEEK_API_KEY 等）> `OPENAI_API_KEY` 兜底

**为什么有兜底**：多数 OpenAI-compatible 服务都支持 `OPENAI_API_KEY` 作为通用名（v2 钩子文档 [OpenRouter/Anthropic 等](https://github.com/ultraworkers/claw-code/blob/main/docs/local-openai-compatible-providers.md)）。

## 4. 与 claw-code 架构的偏差说明

D4.1.0 已说明 Python 端没有直接调 `chat/completions` 的参考代码（claw-code 主架构是 Rust）。D4.1.1 实施细节（httpx 客户端 + OpenAI 协议响应解析）属于通用知识，**不依赖 claw-code 内部代码**。

**仅借鉴 3 条架构原则**（来自 D4.1.0 报告 [§参考来源](D4.1-LLM路由层完成.md)）：

- **Capability-Aware** — capability.py 已落地，**D4.1.1 不变**
- **Provider-Agnostic 抽象** — providers.py 已落地，**D4.1.1 新增 4 类异常分层，符合"基类宽、子类窄"原则**
- **Failure-Stop with Circuit Breaker** — fallback.py + router.py 已落地，**D4.1.1 异常 → fallback 链路验证**（`TestChatIntegrationWithRouter` 端到端测试）

**故意不照搬的 claw-code 模式**：

- claw-code Rust 端用 trait + async，**本项目用 dataclass(frozen=True) + 同步**，原因：Python 端用户量小（个人 AI 员工），同步更易调试
- claw-code 用 `axum` 风格的 router 中间件做 fallback，**本项目用 dataclass 链 + CircuitBreaker**，原因：简单场景下 dataclass 已足够
- claw-code 不区分业务异常和编程错误（Rust 用 `Result<T, E>`），**本项目显式分层**，原因：参考 D3.3.3 教训，Python 异常更易误用
