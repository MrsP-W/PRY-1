## 3. 误报率深度分析

### 3.1 new_merchant 100% 误报根因

```
faker 样本中商家分布(每个 1-2 笔):
  微信 2024: 星巴克×2 / 美团 / 工资 / 滴滴
  微信 2025: 麦当劳×2 / 沃尔玛 / 兼职 / 肯德基
  微信 2026: 瑞幸×2 / 美团 / 工资 / 滴滴
  支付宝 2024-2026: 类似(共 10+ 商家)
```

**根因**:
- `MIN_HISTORY_FOR_PROFILE = 5`(商家画像冷启动阈值)
- 每个商家只有 1-2 笔历史,远低于 5 笔阈值
- → 全部触发 new_merchant 标记(业务信号,但被作为"异常"列入)

**真实业务解读**:
- `new_merchant` 实际是"冷启动业务信号",**不是真异常**
- 当前实现把 new_merchant 当异常计入 results,但语义上它是"无画像标记"
- 月报里应该区分"冷启动" vs "真异常",不要都列在异常告警段

### 3.2 frequency_5tx_per_hour 67% 误报根因

**根因**:
- `HOURLY_TX_THRESHOLD = 5`(1 小时 > 5 笔触发)
- 微信 2024-05 5 笔集中在 5/12-5/16(每天 1 笔)
- spike 的 detect_frequency_anomaly 用 `tx.imported_at_ms` 算时窗,样本时序跨度大但单日密集

**实测时窗分析**:
```
微信 2024-05 5 笔:
  5/12 14:30 (38517000 ms) → 5/13 09:15 (42549000 ms) → 5/14 18:45 (50670000 ms)
  5/15 20:00 (56688000 ms) → 5/16 12:30 (62100000 ms)
时窗 1 小时(3600s)内: 0 笔(笔间 > 1 小时)
```

**实际不应该触发**!让我看 detect_frequency_anomaly 代码 — `recent_in_hour = [r for r in recent if r.imported_at_ms >= hour_ago]`,但是 `recent = self._tx_store.list_by_source(tx.source, since=since_date, limit=200)`,而 `since_date = (datetime.fromtimestamp(hour_ago / 1000) - timedelta(days=1)).date()`。

问题:`since=since_date` 是 date 类型,而 `Transaction.transaction_date >= since_date` 是 date 比较(只比日期,不比时间)。所以 5/16 12:30 的笔会把 5/12 14:30 也查出来(因为 5/12 >= since_date = 5/15),然后 5 笔全在 `recent_in_hour` 中(因为所有 imported_at_ms >= hour_ago = 5/16 11:30)。

**Bug 根因**:`list_by_source` 用 date 过滤,但 `detect_frequency_anomaly` 期望的是"过去 1 小时"的精确时窗。这是 `frequency_5tx_per_hour` 误报的真因。

## 4. 2 个核心问题(需修复)

### 问题 1:new_merchant 语义混淆

**现状**:Detector 把 < 5 笔历史返回 None 走 new_merchant 标记,列入 AnomalyResult。
**问题**:new_merchant 是"冷启动业务信号",不是"真异常"。
**建议修复**(D8.5 待规划):
- 把 new_merchant 从 `results` 列表移除(不计入异常统计)
- 或:新增 `signals` 字段(冷启动/异常分离,沿 D4.7.3 v1.0.1 业务阻断 vs 技术失败范本)
- 月报 `{anomaly_highlights}` 段:不列 new_merchant,改 `{cold_start_merchants}` 段

**优先级**:🟡 中(影响月报可读性 + 用户感知)

### 问题 2:frequency_5tx_per_hour 时窗过滤 bug

**现状**:`detect_frequency_anomaly` 用 date 过滤导入时窗,实际应该是精确时间窗。
**问题**:date 过滤导致跨天单笔被算成同小时,触发频率异常。
**建议修复**(D8.5 待规划):
- 新增 `TransactionStore.list_by_source_in_time_window(source, since_ms, until_ms, limit)`
- Detector 改调精确时间窗接口
- 单元测试:5 笔跨天但单日 ≤ 5 笔,期望不触发

**优先级**:🚨 高(直接导致 67% 误报,影响核心业务)
