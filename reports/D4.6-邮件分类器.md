## 3. ⚡ 性能(估算,未 spike)

D4.6 邮件分类单次调用包含 5 步(对齐 D4.5 范本):

| 步骤 | 估算耗时 | 说明 |

| **总端到端** | **~500-3000ms** | LLM HTTP 主导,policy 叠加 < 1% |

**结论**:**policy 层叠加 < 1ms / classify**,D4.6 LLM 性能不受影响。**D4.6.1+ 在 1 千封真实邮件 spike 验证**(同 D4.5 §3 模式)。

## 4. ⚠️ 已知限制(D4.6.1+ 复检 P 项,5 段待人工审批)

### 4.1 限制 1:5 类是硬编码,不支持自定义

- **现状**:`EmailCategory` 5 类(URGENT/TODO/FYI/SPAM/PERSONAL)硬编码在 StrEnum
- **风险**:**用户邮件场景可能需要更多类**(如 NEWSLETTER / INVOICE / RECEIPT / MEETING 等)
- **缓解**:D4.6.1+ 改 5 类 → 6+ 类(NEW: NEWSLETTER 单独一类,INVOICE 由 D4.7+ drafter 接管)
- **决策**:✅ **接受现状** — 5 类是 week 1 MVP 最小集,与 LLM prompt 一致;扩到 6+ 类是 D4.6.1 复检项

### 4.2 限制 2:`classify_batch` 是顺序串行,不支持并发

- **现状**:`classify_batch` 顺序串行调 `classify`,避免触发熔断 / 雪崩
- **风险**:**100 封邮件顺序分类 ≈ 50-300s**(每封 500-3000ms)
- **缓解**:D4.6.1+ 用 `asyncio.gather` + httpx async 改并发,目标 100 封 < 30s
- **决策**:✅ **接受现状** — D4.6 顺序串行是稳健优先(避免熔断/雪崩),并发留 D4.6.1+;Week 2 业务层调方一般 < 10 封,延迟可接受

### 4.3 限制 3:业务字段透传口径锁定 5 项

- **现状**:`EmailClassifierAdapter.classify_and_emit` 固定透传 `category / confidence / model_full_id / email_id / source` 5 项到 event_metadata
- **风险**:**D4.7+ drafter / classifier_v2 想透传新字段**(如 drafter 主题、分类历史)需扩 `extra_business_payload` 协议
- **缓解**:`PolicyEngine._emit_decision_event` 的 `extra_business_payload` kwargs 已通用化(任意 key 可传),D4.7+ 直接复用

**v1.0.2 修复方案**(决策:**拆双入口 + type system 杜绝状态耦合**):

1. **P1-1 架构级修复**:`classify_and_emit` 拆为双入口,从 type system 杜绝状态耦合
   - **成功入口** `classify_and_emit(*, email_id, classification, transport_alive, run_id, now_ms)` — 无 `last_classify_failed` / `consecutive_classify_failures` 参数,强制 `cf=0, last_classify_failed=False` → 永不触发 retry/escalate
   - **失败入口** `record_classify_failure_and_emit(*, email_id, last_error, consecutive_classify_failures, transport_alive, run_id, now_ms)` — 必填 `last_error` + `cf >= 1`,隐式 `last_classify_failed=True`,synthetic conf=0 → AC[0]=False → 永不触发 merge
   - 新增 `build_classify_failure_packet` factory(失败专用 8 字段 packet,objective 前缀 `email_classify_failed`)
2. **P1-2 严判补全**:`classify_and_emit` 严判段加
   - `category_value not in _VALID_CLASSIFY_CATEGORIES`(5 类 frozenset 内联常量)→ ValueError
   - `latency_ms < 0` 拒收 → ValueError(防御时钟回退)
3. **P2-3 JSON 提取升级**:拆为 `_find_all_balanced_json(raw)`(收集所有平衡 JSON 块)+ `_extract_balanced_json(raw)`(遍历,选第一个 json.loads 成功 + 是 dict + 同时含 `category` + `confidence` 字段的;兜底返回第一个平衡 JSON)
4. **P2-4 批处理契约补全**:`classify_batch` type hint 从 3 类扩到 5 类 `list[ClassificationResult | ClassifierResponseError | LLMError | ValueError | KeyError]`;`missing_keys = [k for k in ... if k not in email]` 检测缺字段,入 `KeyError(f"emails[{i}] 缺字段 {missing_keys}")` 到 results,不外抛
5. **P2-5 factory 拒 NaN**:`build_classify_packet` 加 `if not math.isfinite(confidence): raise ValueError(f"confidence 必须是有限数字(非 NaN/Inf), 实际 {confidence}")`,在 0-1 范围检查前
