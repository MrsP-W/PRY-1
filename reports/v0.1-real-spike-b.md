### 2.2 L1 幂等关键发现(2026-06-15)

**实测**: 第二次跑相同 `--n 1` 显示 `inserted=0 skipped=1` —— **L1 UNIQUE 幂等完美工作**。

- **根因**: `NoteStore.insert` 内 `try/except IntegrityError` 兜底 → 转 `NoteDuplicateError`(沿 D3.3.2 教训 + D9.6.3 P2-1)
- **语义**: 已同步过的笔记归 `skipped`,不算 `failed`,整批 exit 0
- **价值**: W3 真跑时,如果同一条笔记已同步过(测试期间多次按 ⌥⌘N 重跑),不会污染 DB

### 2.4 清理 spike 残留(关键)

```python
# 用 NoteStore + sessionmaker + Note.delete() 清理 spike-folder
from sqlalchemy import delete
with sf() as s:
    res = s.execute(delete(Note).where(Note.folder == 'spike-folder'))
    s.commit()
# 清理后 notes count: 0 (生产 DB 干净)
```

**沿用 NoteStore API**: `store = NoteStore(session_factory)`,**没有** `store.session()` 方法,直接用 `sf()` contextmanager(D9.1 范本)。

### 2.5 教训

- **dry-run 不等于 0 影响**: 即使 spike 跑 faker,也会写真实 DB(默认 DB 路径)。**必须清理**或用 `--db-path <test.db>` 隔离(沿 D3.1 范本)
- **`sync_notes.py` 已补真跑安全阀**: `sync` 需 `NOTES_REAL_NETWORK=1`,支持 `--max-rows N` 限量;`sync` / `spike` 均支持 `--db-path` 隔离。`spike` 仍是唯一非生产写入方式。
- **L1 幂等实测通过**: 这是 Spike B 真跑的安全网(重复 sync 不污染)
