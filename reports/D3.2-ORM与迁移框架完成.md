## 2. 关键设计决策

### 2.1 ⭐ row_factory 推到方法入口（核心架构决策 — D3.2 最大调整）

**问题**：D3.1 决策在 `Database.open()` 一次性设 `conn.row_factory = _dict_factory`，业务代码 `db.fetch_one()` 拿 dict 很爽。但 D3.2 引入 SQLAlchemy 后：

```
SA 2.0 SQLite dialect → first_connect 钩子 → get_isolation_level("ROLLBACK")
→ cursor.fetchone()[0]   # 期望 tuple-like（res[0] 索引）
→ conn.row_factory = _dict_factory 会让 row[0] 抛 KeyError: 0
```

**失败的尝试**（按时间顺序）：

1. ❌ **尝试 1**：在 SA creator 回调里 `conn.row_factory = None`（覆盖）
   - 副作用：`db.fetch_all()` 业务代码 row 变 tuple（不是 dict），破坏 D3.1 接口
2. ❌ **尝试 2**：monkey-patch sqlcipher3.Cursor 加 `__enter__/__exit__`
   - 调查发现：SA 2.0 SQLite dialect **不**调 `with cursor()`（只有 Oracle dialect 调）— 这个 patch 不需要

**✅ 最终方案**：把 `row_factory` 推到方法入口

```python
# core/db.py
def fetch_one(self, sql, params=()):
    self._conn.row_factory = _dict_factory   # 临时设
    cursor = self._conn.execute(sql, params)
    try:
        return cast(dict[str, Any] | None, cursor.fetchone())
    finally:
        self._conn.row_factory = None        # 立即还原

# Database.open() 里 **不**再设 conn.row_factory
# conn.row_factory 常态是 None → SA dialect 探针天然 OK
# db.fetch_*() 方法入口临时设 → 业务代码仍拿 dict
```

**为什么 cursor.fetchall/fetchone 在 row_factory 还原后仍能拿 dict**：
- `cursor` 在 `execute()` 时**已经绑定** dict_factory（cursor 内部存了 row_factory 引用）
- `cursor.fetchall()` 实际调 row_factory 是 cursor 内部状态，**不**每次问 conn
- 所以 `finally: row_factory = None` 还原的是 conn 状态，cursor 内部的 row_factory 不受影响

**关键认知**：dict_factory 的"绑定时机"是 cursor 内部状态，不是 conn 状态。**Why**：这是 D3.1 决策错误的根因 — 把 row_factory 设在 conn 而不是 cursor 维度，导致 SA dialect 探针冲突。**How to apply**：以后任何 `db.execute()` / `db.fetch_*()` 类方法都按 "**临时设 + 立刻还原**" 模板写。

### 2.2 ⭐ server_default 必须配合 default=（D3.2 必踩坑）

**问题**：仅设 Python-side `default=` 不够 — SA 生成的 DDL 不会有 SQL DEFAULT 子句：

```python
# ❌ 错误：仅 Python default
class SyncState(Base):
    last_status: Mapped[str] = mapped_column(Text, nullable=False, default="pending")
    # SA 生成 DDL:  last_status TEXT NOT NULL        # 没有 DEFAULT
    # → raw SQL INSERT 不传 last_status → "NOT NULL constraint failed"

# ✅ 正确：default + server_default 配对
class SyncState(Base):
    last_status: Mapped[str] = mapped_column(
        Text, nullable=False, default="pending", server_default="pending"
    )
    # SA 生成 DDL:  last_status TEXT DEFAULT 'pending' NOT NULL
```

**踩坑现场**：D3.2 测试跑 `Base.metadata.create_all` + 直接 `session.add(SyncState(source="qq", updated_at=3000))` → 抛 `IntegrityError: NOT NULL constraint failed: sync_state.last_status`。**根因**：`default=` 是 ORM 层 Python 默认值（commit 时 SA 自动填），但当 SA 在 `create_all` 时生成 DDL，只看 `server_default=` 才写 SQL DEFAULT 子句。

**Why**：D3.1 schema.sql 有 `DEFAULT 'pending'`，D3.2 ORM 必须 mirror 这点，否则两套 schema 漂移。**How to apply**：每个有 `default=` 的 Model 字段**必须**配 `server_default=`（TEXT → `""` / `[]` / `"pending"`；INTEGER → `"0"`）。

### 2.3 alembic env.py 走 D3.1.2 受控 `connection` property

**D3.1.2 已设计** `Database.connection` property — alembic 拿受控 raw connection 入口，**不**直接碰私有 `_conn`：

```python
# src/my_ai_employee/core/migrations/env.py
def run_migrations_online():
    from my_ai_employee.core.db import Database
    db = Database.open()
    creator = make_sqlalchemy_creator(db)  # 走 db.connection（D3.1.2 受控入口）

    configuration = config.get_section(config.config_ini_section, {})
    configuration["sqlalchemy.url"] = "sqlite:///"  # 占位
    connectable = engine_from_config(
        configuration, prefix="sqlalchemy.",
        poolclass=pool.NullPool, creator=creator,
    )
    with connectable.connect() as connection:
        context.configure(
            connection=connection, target_metadata=target_metadata,
            render_as_batch=True,   # SQLite 限制：ALTER TABLE 走 batch
        )
        with context.begin_transaction():
            context.run_migrations()
```

**关键设计**：
1. **不**直接用 `db._conn`（封装泄漏）— 走 `make_sqlalchemy_creator` → 内部调 `db.connection`
2. **`render_as_batch=True`** — SQLite 不支持 `ALTER TABLE` / `DROP COLUMN`，alembic 必须用 batch mode（先 create 新表 + copy 数据 + drop 旧表 + rename）
3. **`poolclass=pool.NullPool`** — alembic 一次性连接，不需要连接池
4. **offline 模式**（`alembic upgrade --sql`）走 SQLAlchemy 字符串渲染，**不**调 creator

### 2.4 sqlcipher3 Cursor 不支持 `with` 上下文管理器 — 但 SA 2.0 不需要

**调查**：尝试 `with db.connection.execute("...") as cursor:` 抛：

```
TypeError: 'sqlcipher3.dbapi2.Cursor' object does not support the context manager protocol
```

**根因**：sqlcipher3 的 Cursor 是 immutable built-in type（`__dict__` 不可写），无法 monkey-patch `__enter__/__exit__`。

**结论**：SA 2.0 SQLite dialect **不**使用 `with cursor()` 模式（grep SA 源码确认）— 只有 Oracle dialect 调 `with cursor()`。所以**不需要**任何 patch。

**Why**：这是个伪雷区 — 真正的痛点是 `with cursor()` 写 smoke test 时碰到的，与 SA 集成无关。**How to apply**：在 D3.2 + 之后的代码里，**不要**写 `with conn.execute(...) as cur:`，直接 `cur = conn.execute(...)` 即可。

### 2.5 ⭐ D3.2.3：JSON 字段用 `JSONList TypeDecorator`（完全 mirror D3.1 schema.sql）

**D3.2 v1.0 决策**：`Email.recipients` / `Email.labels` 用 SA 内置 `JSON` 类型，**DDL 走 JSON**（SQLite 存 JSON 文本）。

**D3.2.3 修复**：D3.1 schema.sql 用 **`TEXT DEFAULT '[]'`**，ORM 必须 mirror — 引入 **`JSONList` TypeDecorator**：

```python
# src/my_ai_employee/core/models.py
class JSONList(TypeDecorator):
    """list ↔ JSON 文本（D3 阶段 mirror schema.sql TEXT DEFAULT '[]'）。"""
    impl = Text
    cache_ok = True

    def process_bind_param(self, value, dialect):
        if value is None:
            return None
        return json.dumps(value, ensure_ascii=False)

    def process_result_value(self, value, dialect):
        if not value:  # None 或空字符串 → []
            return []
        return json.loads(value)


# 用法
recipients: Mapped[list[str]] = mapped_column(
    JSONList, nullable=False, default=list, server_default="[]"
)
```

- **DDL**：`recipients TEXT DEFAULT '[]' NOT NULL`（完全 mirror schema.sql）
- **写**：`Email(recipients=["a@b.com"])` — `process_bind_param` 序列化为 JSON 文本
- **读**：`email.recipients` → `["a@b.com"]` — `process_result_value` 反序列化
- **D3.3 用法**：`IMAPConnector.envelope.to` 是 list[str] → 直接 `Email(recipients=list)` 即可
- **中文支持**：`ensure_ascii=False`（"收件箱" / "重要" 不转义成 `\u...`）

**Why**：D3.1 schema.sql 是真理之源（更接近 SQL 标准、SQLite CLI 直接看得懂），ORM 不应偏离。**How to apply**：以后任何 list/dict 字段都走 `JSONList`（或对应 TypeDecorator）+ `server_default="[]"`。

### 2.6 ⭐ D3.2.3：Label.name COLLATE NOCASE — `sa.Text(collation="NOCASE")`

**D3.2 v1.0 错误**：

```python
# ❌ alembic upgrade head --sql 抛 ArgumentError
op.create_table(
    "labels",
    sa.Column("name", sa.Text(), nullable=False, sqlite_collation="NOCASE"),
)
# → sqlalchemy.exc.ArgumentError: Argument 'sqlite_collation' is not accepted by dialect 'sqlite'
```

**根因**：`sqlite_collation` 是 SA 给老版本准备的 column 参数，但 SA 2.0 已统一到类型参数 `collation=`，且方言层不再接受 column 维度的 `sqlite_collation`。

**D3.2.3 修复**：collation 是**类型参数**，不是 column 参数：

```python
# ✅ models.py
name: Mapped[str] = mapped_column(
    Text(collation="NOCASE"),    # ← collation 在类型上
    nullable=False,
)

# ✅ 0001_initial.py
sa.Column("name", sa.Text(collation="NOCASE"), nullable=False)
```

**Why**：SA 2.0 重构了方言参数校验 — column 维度只接受 `nullable / default / server_default / primary_key / autoincrement / unique / index / comment / info`，collation 必须在类型上声明（这样 SQLite/Postgres/MySQL 方言都能识别）。**How to apply**：D4+ 任何列级 collation 修饰符都走 `Text(collation=...)` / `String(collation=...)`。

### 2.7 ⭐ D3.2.3：DESC 索引用 `text("col DESC")` 表达

**D3.2 v1.0 错误**：D3.1 schema 是 `received_at DESC`，D3.2 v1.0 写 `Index("idx_emails_received_at", "received_at")` — 升序索引。

**D3.2.3 修复**：

```python
# ✅ models.py
from sqlalchemy import text
Index("idx_emails_received_at", text("received_at DESC"))
Index("idx_emails_source_received", "source", text("received_at DESC"))
Index("idx_audit_log_created_at", text("created_at DESC"))

# ✅ 0001_initial.py
op.create_index("idx_emails_received_at", "emails", [sa.text("received_at DESC")])
```

**验证**：DDL 输出 `CREATE INDEX idx_emails_received_at ON emails (received_at DESC);`（`test_migrations.py::test_alembic_creates_desc_indexes` 验证）。

**Why**：D3 阶段热路径是"按时间倒序取最近邮件/审计事件" — 升序索引无法 service 这个查询模式。**How to apply**：D4+ 加新索引时，凡热路径涉及时间排序默认 DESC。

## 4. 已知雷区 + 解决方案（D3.2 新增 4 个 + D3.2.3 新增 4 个 — **共 8 个**）

| # | 雷区 | 触发 | 解决 | 闭环 |
|---|------|------|------|------|
| 1 | **SA dialect 探针 KeyError: 0** | D3.1 在 `Database.open()` 设 `conn.row_factory = _dict_factory` | **推到方法入口**：`db.execute/fetch_*` 内临时设，**finally 立刻还原**为 None | D3.2 ✅ |
| 2 | **server_default 缺失导致 NOT NULL** | ORM `default="pending"` 不生成 SQL DEFAULT 子句 | **default + server_default 配对**：每个有 Python 默认值的字段都加 `server_default=` | D3.2 ✅ |
| 3 | **sqlcipher3 Cursor 不支持 `with`** | `with conn.execute(...) as cur:` 抛 TypeError | SA 2.0 SQLite dialect 不调 `with cursor()` — **不需要 patch**，直接 `cur = conn.execute(...)` | D3.2 ✅ |
| 4 | **D3.1.2 connection property 测试断言错** | row 从 dict 变 tuple（row_factory 调整） | 改 `row[0].lower() == "wal"`（解构 tuple）| D3.2 ✅ |
| 5 | ⭐ **NOCASE 写法 ArgumentError** | `Column(..., sqlite_collation="NOCASE")` 报 `'sqlite_collation' is not accepted by dialect 'sqlite'` | **collation 是类型参数**：`sa.Text(collation="NOCASE")`（不是 column 参数）| **D3.2.3 ✅**（`alembic upgrade head --sql` exit 0）|
| 6 | ⭐ **JSON 字段未 mirror schema.sql** | D3.2 v1.0 migration 走 `sa.JSON()`，schema.sql 走 `TEXT DEFAULT '[]'` — 两套 schema 漂移 | **JSONList TypeDecorator**：DDL 走 TEXT，ORM 走 list，**完全 mirror** | **D3.2.3 ✅**（DDL 与 schema.sql 1:1）|
| 7 | ⭐ **DESC 索引缺失** | D3.1 schema 是 DESC，D3.2 v1.0 是普通升序 — 热路径"按时间倒序"无索引支持 | **`text("col DESC")`**：ORM 端用 `Index(..., text("col DESC"))` | **D3.2.3 ✅**（DDL 含 DESC）|
| 8 | ⭐ **EmailLabel 关系测试覆盖不足** | D3.2 v1.0 文档说 "Email.labels M-N via EmailLabel"，实际 `Email.labels` 是 JSON 字段，关系是 `email_labels` | **保留双轨**：JSON 字段（快速过滤）+ EmailLabel 关系（规范化 join），**测试覆盖双轨** | **D3.2.3 ✅**（3 关系测试）|

**D3.1 雷区继承**（不变）：
- PRAGMA 不支持 `?` 占位符 → `executescript` + 单引号转义
- SQLCipher 懒校验 → `PRAGMA quick_check` 主动触发
- sqlcipher3 Cursor 非 sqlite3.Cursor 子类 → 自定义 `_dict_factory`

## 9. 关键认知汇总（D3.2 教学版）

### 9.1 适配层 vs 改源码的边界

- **改 sqlcipher3 源码**（monkey-patch Cursor 加 `__enter__`）：❌ 不可行（immutable built-in）
- **改 SA dialect**：❌ 不可行（破坏 SA 升级路径）
- **✅ 适配层**（`sqlcipher_compat.py`）走 creator 模式：可逆 + 可升级 + 测试覆盖

### 9.2 ORM 与手写 SQL 的边界

- **ORM 适合**：业务实体（Email / Label）+ 关系维护 + 类型安全
- **手写 SQL 适合**：性能敏感 / 复杂 join / 元查询（PRAGMA / sqlite_master）
- **D3 阶段决策**：混合模式 — Model 走 ORM，PRAGMA 走 `db.fetch_one()`

### 9.3 alembic 何时介入

- **D3.2 已经 alembic init + 首次迁移**（0001_initial.py）— 真 alembic 迁移测试（tests/core/test_migrations.py）已闭环
- **D4 起**：业务 schema 变更走 `alembic revision --autogenerate -m "..."` + `alembic upgrade head` + 跑 test_migrations.py 验证
- **D3.3 决策**：alembic 已就绪，sync 脚本走 ORM + `Base.metadata.create_all`（D3 阶段保持 init_schema 兼容路径）

### 9.4 ⭐ D3.2.3 教学：TypeDecorator 的设计哲学

- **D3.2 v1.0 用 SA 内置 JSON**：方便但 DDL 与 D3.1 schema.sql 漂移
- **D3.2.3 用 JSONList TypeDecorator**：DDL 走 TEXT（mirror schema.sql），ORM 走 list（类型安全）
- **设计原则**：**DDL 真理之源是 schema.sql**（SQLite CLI 直接看得懂），ORM TypeDecorator 只在 marshalling 层做转换
- **D4+ 启发**：其他 list/dict 字段都按这个模式 — 先看 schema.sql 写什么类型，TypeDecorator 透明处理 Python ↔ DB 转换

### 9.5 ⭐ D3.2.3 教学：SA 2.0 column 参数 vs 类型参数

- **column 参数**：`nullable / default / server_default / primary_key / autoincrement / unique / index / comment / info`
- **类型参数**：`collation / length / precision / scale / charset / collation` 等方言相关
- **D3.2.3 教训**：写 `sa.Column("name", sa.Text(), sqlite_collation="NOCASE")` 失败，因为 `sqlite_collation` 不在 column 维度白名单
- **正确**：`sa.Column("name", sa.Text(collation="NOCASE"))` — collation 在类型上
- **D4+ 启发**：写新 Model 时，先确认字段类型是 `Text` / `String` / `Integer` / `JSON` / `DateTime`，再考虑类型参数；column 参数只关心"这列的元数据"。
