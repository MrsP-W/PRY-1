## 4. 撞坑 4 个(#71-#75)

### 撞坑 #71 · OutboxStatus 大小写不匹配(P0 业务错)

**触发**:Stage 4 第一笔 `outbox.approve` 返回 409 + `ValueError: status 必须是 OutboxStatus 6 选 1 frozenset({'sending', 'approved', 'cancelled', 'pending_send', 'sent', 'failed'}),实际 'APPROVED'`。

**根因**:`business_writer_impl.py:433` 传 `new_status="APPROVED"`(大写) + `from_status="PENDING_SEND"`,但 `OutboxStatus` StrEnum 枚举值是小写。源出于实现时没跟 enum 契约对齐。

**修法**:
- 改 `business_writer_impl.py:433` `new_status="APPROVED"` → `"approved"`,`from_status="PENDING_SEND"` → `"pending_send"`
- 改 `business_writer_impl.py:479` 同样改 `cancel_outbox` 的 `new_status` + `from_status` 小写
- 同步改 `test_business_writer_impl.py::test_real_write_passes_last_approved_at_ms` + `::test_real_write_calls_update_status_with_none_approved_at` 2 个契约测试的 `assert ==` 改小写
- `reason` 字段文本保持大写(用户友好消息,不需跟 enum 严格一致)

**Why 漏测**:
- v0.2.55 提前接地的 6 个新契约测试只测 dry-run/raise,**没覆盖真实 call 路径**
- 5 门全开 spike 是发现 #71 的唯一手段

**Why 选 A 方案(热修小写)**:
- 最小改动(2 行 + 2 测试断言改 4 个常量)
- 与 `OutboxStatus` 枚举契约对齐,无副作用
- 保留 `reason` 字段大写(用户友好)

### 撞坑 #72-#75 · spike 限定 4 坑堆叠

详见 [[pitfall-72-75-spike-threading-pitfalls]]。修法:**spike 实写绕开 HTTP,主线程直调 writer**。

| # | 错误 | 修法 |
|---|------|------|
| 72 | `ProgrammingError: SQLite objects created in a thread can only be used in that same thread` | 实写绕开 HTTP |
| 73 | `urlopen TimeoutError`(handle_request 时序) | 回退 ThreadingHTTPServer |
| 74 | `Cannot operate on a closed database`(NullPool + Database 单例 close) | 恢复标准 Pool + 主线程 |
| 75 | `ValueError: 状态机守卫拒绝非法转换: 'NEW' → 'ARCHIVED'` | seed_note 改 `sync_status="STRUCTURED"` |

**Why 是 spike 限定**:
- 生产环境 launchd 单线程跑 handler,Database 单例 `_conn` 安全
- 撞坑 #72-#74 是 ThreadingHTTPServer 模式(本 spike 限定)与 Database 单例模型不兼容
- 撞坑 #75 是 Note 状态机安全护栏正常生效(数据准备问题)
