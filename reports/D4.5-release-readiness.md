## 3. ⚡ 性能（D3.3 spike 复测,集成层叠加开销）

### 3.1 D3.3 IMAPSync 基线（复用）

- **基线**:`scripts/sync_imap.py` + `core/sync.py IMAPSync.run_once()` 1 万封 spike **0.30s / 25921 封/秒**（D3.3.3 复测）
- **SQLCipher 加密** + 100/批 commit ORM + `SyncState` upsert + 失败隔离

### 3.2 D4.5 集成层叠加（评估,未 spike）

D4.5 `evaluate_and_emit` 单次调用包含 5 步:

| 步骤 | 估算耗时 | 说明 |
|------|----------|------|
| 1) `build_imap_sync_packet` | < 0.01ms | 纯 dataclass 构造,8 字段赋值 |
| 2) `build_sync_policy_context` | < 0.01ms | 12 字段 dict 构造,无 IO |
| 3) `PolicyEngine.evaluate` (含 6 决策) | < 0.5ms | 6 rule 函数 + 1 event insert(D4.3 复用,fingerprint 缓存命中) |
| 4) `LaneBoard.record_to_lane` | < 0.1ms | in-memory dict.get / add / update |
| 5) `Heartbeat.tick_heartbeat` | < 0.1ms | in-memory state + now_ms 比较 |
| **合计** | **< 1ms** | 比 D3.3 IMAP sync 单批(100 封 ~30ms)低 30 倍 |

**结论**:**集成层叠加 < 1ms / sync**,D3.3 1 万封 / 30s spike 性能不受影响。**D4.5.1+ 在 1 万封真实 spike 实测验证**（详见 §4.3 审批项 3）。

### 3.3 Memory 叠加

- `SyncPolicyAdapter` 实例 4 依赖（`event_store` / `engine` / `heartbeat` / `board`）+ 1 字符串 `source`
- 估算 < 1KB / 实例,业务层可单例复用
- 每次 `evaluate_and_emit` 不创建新对象(在 adapter 实例上调用),GC 压力零

## 4. ⚠️ 已知限制（D4.5.1+ 复检 P 项,5 段待人工审批）

### 4.1 限制 1:6 决策是声明式,`evaluate_and_emit` 不替 caller 执行

- **现状**:`evaluate_and_emit` 只 emit 事件 + 推进 lane + 刷新 heartbeat,实际 retry/merge/escalate 由 D5+ 业务调度器决定
- **风险**:**D4.5 报告后,若 caller 直接把 6 决策丢一旁,可能导致 "MergeRequired 触发但没 merge,merge 数据丢失"**
- **缓解**:`PolicyEvaluation.decisions` 暴露 `priority / rule_name / kind / explanation`,caller 可遍历执行（留 executor pattern 给 D4.5.1+）
- **决策**:✅ **接受现状** — D3.3.3 异常窄化教训应用,policy 层不替 caller 决定;D4.5.1+ 加 `executor` pattern(retry 调 `IMAPSync.run_once` / escalate 写 events 表 escalation row)

### 4.2 限制 2:LaneBoard in-memory 仍未持久化

- **现状**:`LaneBoard` 是 `dict[entry_id, LaneEntry]` in-memory 状态,跨进程不可见
- **风险**:**重启后 lane 状态清零**,若要"看历史 7 天 lane 状态"必须从 events 表 event_metadata 推算
- **缓解**:`record_to_lane` 推进状态时,D4.5 已隐式通过 `PolicyEngine._emit_decision_event` 落 `POLICY_DECISION_MADE` 事件(events 表有 `lane_entry_id` / `run_id` / `all_decisions` 字段,**v1.0.1 反馈修复后已显式写入**,可推算)
- **决策**:✅ **接受现状** — D4.5.1+ 落 `lane.entry.added` / `status_changed` 事件,允许 `mmx policy status --history` 子命令查询

### 4.3 限制 3:无 1 万封真实 spike

- **现状**:D4.5 集成测试 35 个全在**单次 sync 调用级别**,未跑 1 万封真实邮件端到端
- **风险**:**D3.3 IMAPSync spike 0.30s 是基线**,但 D4.5 evaluate_and_emit 叠加 < 1ms 是估算,未在 1 万封真实数据上验证
- **缓解**:D3.3 IMAP 1 万封 spike 6/8 已验证,D4.5 集成层只增加 < 1ms / sync,理论影响 < 1%
- **决策**:⏳ **D4.5.1+ 必跑** — 在 1 万封真实邮件上跑 `evaluate_and_emit` 30 天,验证 (a) 集成层叠加 < 5% 性能开销 (b) events 表 1 万行查询 < 1s (c) fingerprint dedupe 命中率 > 95%(同业务事件多次重试)

### 4.4 限制 4:CLI 集成未做

- **现状**:`LaneBoard.to_status_json()` / `PolicyEngine.evaluate().to_status_json()` 状态 JSON 导出方法已实现,但 CLI 子命令 `mmx policy status` / `mmx policy history` 未集成
- **风险**:**用户看不到 lane / decision 状态**,只能写 Python 脚本调 API
- **缓解**:D4.5 暴露完整 API,业务层 + 未来 Web UI 可直接调
- **决策**:✅ **接受现状** — D4.5.1+ 加 `mmx policy status` / `mmx policy history` 子命令,用 `click` 或 `argparse` 集成
