## 3. 设计决策(沿 0013 范本无 FK 软标记)

### 3.1 初版 0014 失败根因

初版 0014 包含 `op.create_foreign_key` self-reference (notes.id → notes.candidate_match_id),跑 alembic upgrade head --sql 报错:

```
NotImplementedError: No support for ALTER of constraints in SQLite dialect.
Please refer to the batch mode feature which allows for SQLite migrations using a
copy-and-move strategy.
```

### 3.2 修复策略:沿 0013 范本无 FK 软标记

| 维度 | 初版设计 | 修复后(沿 0013) |
|------|----------|------------------|
| candidate_match_id 字段类型 | `ForeignKey("notes.id", ondelete="SET NULL")` | `Integer, nullable=True`(纯字段无 FK) |
| 引用一致性保护 | DB 层 FK | 应用层 NoteStore.insert 派生时 `select Note.id` 校验 |
| 迁移 DDL | ADD COLUMN + CREATE FK(2 步,batch mode 限制) | ADD COLUMN(1 步,干净) |
| 复杂度 | 高(SQLite batch mode + 自引用 FK + 临时表重建) | 低(沿 0013 单 ADD COLUMN) |

### 3.3 决策依据

- **0013 范本已定调** "L2 软标记 + 无 FK"(沿 D6.4 transactions L2 范本)
- **L1 UNIQUE(apple_note_id) 已做硬约束**,L2 是软标记层
- **候选 id 存在性**由应用层 NoteStore.insert 派生时 `select Note.id` 校验(已 select,无新增查询)
- **跨源去重的"硬"保护力**由 0013 normalized_fingerprint 索引 + L1 UNIQUE(apple_note_id) 二层提供,candidate_match_id FK 是装饰

## 6. @教练员 沉淀 — Claude Code 技巧

### 6.1 SQLite 自引用 FK 在 alembic 中的限制(本轮发现)

**问题**:`op.create_foreign_key` 自引用同表 FK 在 SQLite 上不支持,即便配 `render_as_batch=True`。

**陷阱**:
- SQLite `ALTER TABLE ADD CONSTRAINT` 不支持
- alembic batch mode 用 copy-and-move strategy(临时表 + 拷数据 + 删旧表 + 改临时表名)对自引用 FK + 多步迁移行为复杂易错
- 错误信息"NotImplementedError: No support for ALTER of constraints in SQLite dialect" 容易误以为是 batch mode 没配

**解法**:**沿 0013 范本无 FK 软标记**
- 字段纯 Integer,引用一致性由应用层 select 校验
- 跨源去重的"硬"保护力靠 0013 INDEX + L1 UNIQUE 二层
- FK 是装饰,不是必需

**应用场景**:
- 自引用 FK(本表 id → 本表字段)
- 多步迁移中需要重建 FK
- SQLite + alembic 任何带 FK 的 DDL

### 6.2 L2 软标记 + 应用层校验范本(本轮范本可复用)

**范本结构**:
1. ORM 字段:纯 Integer 字段,无 FK
2. 写入时:`select earliest.id where fingerprint=X order by id asc limit 1`
3. 查询时:`where needs_confirm=1` 走热路径 INDEX
4. 应用层一致性:select 校验存在性

**vs L1 UNIQUE 的关系**:
- L1 UNIQUE(apple_note_id)= 硬约束,DB 拒绝冲突
- L2 needs_confirm + candidate_match_id = 软标记,应用层展示 + 用户决策

### 6.3 alembic 迁移设计三选一(本轮决策)

| 方案 | 适用 | 本轮选择 |
|------|------|----------|
| A. ADD COLUMN + FK(自引用) | DB 层强约束 | ❌ SQLite batch mode 限制 |
| B. ADD COLUMN 无 FK + 应用层校验(本轮) | 软标记 + 应用层选最早 | ✅ |
| C. 新表 + FK(自引用) | 完全重设计 | ❌ DDL 改动大,downgrade 难 |

**选 B 理由**:沿 0013 范本,改动小、downgrade 干净、引用一致性由应用层保。
