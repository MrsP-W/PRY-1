### 2.2 修复方案

#### 修复 1 · test_oauth2.py L204 + L222(call-arg)

```python
# 修前
monkeypatch.setattr(
    keychain, "set_password", lambda *args, **kwargs: keychain.KeychainResult(success=True)
)
# 修后
monkeypatch.setattr(
    keychain, "set_password", lambda *args, **kwargs: keychain.KeychainResult(ok=True)
```

**依据**:`src/my_ai_employee/core/keychain.py:47` KeychainResult dataclass 字段契约 = `ok: bool` 而非 `success`。

#### 修复 2 · test_v0_1_s3_outbox.py L53-61(arg-type + union-attr 双重)

```python
# 修前
for d in drafts:
    report = adapter.store_and_emit(
        email_id=d["email_id"],
        subject=d["subject"],
        body=d["body"],
        tone=d["tone"],
        recipient_email=d["recipient_email"],
        run_id=f"e2e-s3-{d['email_id']}",
    )
    assert report.outbox_stored is True  # 严格模式:object 类型 + 联合类型
    assert report.outbox_id is not None and report.outbox_id >= 1  # 严格模式

# 修后
from typing import cast
from my_ai_employee.policy.outbox_adapter import OutboxDecisionReport

for d in drafts:
    report = adapter.store_and_emit(
        email_id=cast(int, d["email_id"]),
        subject=cast(str, d["subject"]),
        body=cast(str, d["body"]),
        tone=cast(str, d["tone"]),
        recipient_email=cast(str, d["recipient_email"]),
        run_id=f"e2e-s3-{d['email_id']}",
    )
    # 窄化联合类型:store_and_emit 可能返回 OutboxDecisionReport | OutboxBlockedDecisionReport
    # 黑名单命中时是 OutboxBlockedDecisionReport(无 outbox_stored / outbox_id)
    assert isinstance(report, OutboxDecisionReport), f"unexpected blocked report: {report}"
    assert report.outbox_stored is True
    assert report.outbox_id is not None and report.outbox_id >= 1
```

**依据**:
- v0.2.23 `cast(int, ...)` 范本(沿用撞坑 #31 历史登记)
- D4.7.3 v1.0.4 P1-1 跨字段校验范本(D4.7.3 v1.0.4)+ D4.7.4 业务层接入 isinstance 守卫

## 4. 撞坑沉淀

### 4.1 撞坑 #55(本轮新增)— 严格模式 mypy 双 0 范本

**现象**:撞坑 #31 历史 dry-run 发现 13 errors,其中 9 errors 是 `mypy --check-untyped-defs` 严格模式专属,默认模式 0 errors。

**根因**:
- 默认 mypy 不检查 untyped function bodies(注释提示:`--check-untyped-defs`)
- 严格模式启用后,dict 访问默认 object 类型 + 联合类型未窄化 + 错误字段名 都会被捕获
- v0.2.23 commit `5920776` docs-only 登记撞坑 #31,但**未启用严格模式**,所以默认 CI 永远 0 errors

**范本**:**严格模式 mypy 双 0 范本** = 默认模式 0 errors + 严格模式 0 errors
- 撞坑 #18 风险门控范本延伸:不只满足 CI 通过,要满足严格模式也通过
- 撞坑 #51 链路逻辑:doc 口径(默认模式 0)必须与实际状态(严格模式 9 errors)一致

**教训**:**沿 v0.2.23 `cast()` 范本 + D4.7.3 isinstance 守卫范本修复严格模式 errors** — 任何"严格模式错误"都应:
1. dict 访问用 `cast(int, ...)` 显式标注类型
2. 联合类型用 `isinstance()` 守卫窄化
3. 字段名错误(`success` vs `ok`)直接修测试
