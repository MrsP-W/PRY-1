## 🎯 状态漂移审查机制设计(4 机制)

### 机制 1:D-step 收官时强制回填

**触发时机**:每个 D-step 子阶段收口 / v0.x.y 阶段收口 / 跨日累计节点。

**实施步骤**:
1. 检查员跑 `git log --since="<阶段起始日>" --pretty=format:"%h %s" | grep -v "^docs\|^test\|^ci\|^chore"` 拿实际 commits
2. 检查员 grep `SESSION-STATE.md` / `MODIFICATION-LOG.md` / `README.md` 拿累计状态
3. diff 两份清单,标记"git log 有但 SESSION-STATE 没"的状态漂移 commits
4. 沿 `7391fe2` docs-only 校准范本提交(1 commit 修多 docs 漂移)

**价值**:把状态漂移撞坑在阶段收口时一次性修掉,避免后期累计漂移。

### 机制 2:每月 1 号 12:00+ 检查员专门做"状态漂移审查"

**触发时机**:每月 1 号 12:00+ 检查员强制截点。

**实施步骤**(沿 [[memory/inspector_plans/]] 范本):
1. 检查员跑 `git log --since="<上月起始日>" --pretty=format:"%h %s"` 拿上月所有 commits
2. 检查员 grep `SESSION-STATE.md` 上月累计状态段
3. diff 两份清单,标记漂移 commits
4. 写 `memory/inspector_plans/<YYYY-MM-DD>.json` plan JSON(沿 v0.2.2 范本)
5. 沿 plan 执行 docs-only 校准(沿 `115fc8e` + `7391fe2` + `5b4a2d8` 范本)

**价值**:7/1 月度复盘准备 + 状态漂移审查机制入库 + 防止漂移累计恶化。

### 机制 3:撞坑恢复 3 步范本(并发进程冲突)

**触发时机**:并发进程实施未 commit + 撞坑发生时。

**实施步骤**(沿 v0.2.2 #8 SMTPProviderFactory 撞坑范本):
1. **步骤 1:撤回自己基于旧状态写的 docs-only 报告** — `rm reports/<冲突相关报告>.md`(避免 stale 报告 commit)
2. **步骤 2:验证并发进程工作通过 8/8 质量门** — pytest + ruff + mypy + alembic + uv build + MD lint + coverage 全绿(8/8 必须全过)
3. **步骤 3:用户授权代 commit 2 commits** — feat commit + docs closure commit 配对(沿 v0.2.2 范本)

**撞坑恢复 commit 范本**:
- commit message 必带"撞坑恢复"关键词 + 报告路径
- 例:`feat(smtp): v0.2.2 #8 SMTPProviderFactory 工厂模式实化(QQ/Outlook/Gmail connector + provider-aware Keychain)` commit message 末尾标注"撞坑记录: 本 commit 由并发进程实施,撞坑后用户授权我代 commit"
- docs closure commit 末尾同样标注撞坑恢复

**价值**:撞坑恢复范本化,后续撞同款问题按 3 步执行,减少决策成本。

### 机制 4:并发进程冲突防撞(每月 1 号 git log vs working tree diff)

**触发时机**:每月 1 号 12:00+ 检查员(与机制 2 同步执行)。

**实施步骤**:
1. 检查员跑 `git status --short | wc -l` 拿 working tree dirty files 数
2. 若 dirty files > 0,跑 `git log --since="<上月起始日>"` 拿上月所有 commits
3. diff commits vs dirty files,标记"未 commit 的 dirty files"(潜在并发冲突)
4. 写入 monthly review report,建议用户手动处理或代 commit

**价值**:每月 1 号强制 git status 检查 + 未 commit 文件清单,提前发现并发进程冲突。

## 🔄 撞坑恢复 3 步范本(详细)

### 步骤 1:撤回自己基于旧状态写的 docs-only 报告

**触发条件**:发现并发进程撞坑(working tree dirty files > 0 + 你之前写的 docs-only 报告基于旧状态)。

**实施**:
```bash
# 撤回自己写的报告
rm reports/<基于旧状态的报告>.md

# 验证撤回
ls reports/<基于旧状态的报告>.md  # 应报错"file not found"
```

**理由**:避免 stale 报告 commit,后续重写更高效。

### 步骤 2:验证并发进程工作通过 8/8 质量门

**触发条件**:撤回报告后,需要验证并发进程工作是否可被接受。

**实施**:
```bash
# 跑 8/8 质量门
uv run pytest --tb=no -q  # 必须 passed
uv run ruff check src tests  # 必须 All checks passed
uv run mypy src  # 必须 0 errors
alembic upgrade head --sql  # 必须 exit 0
uv build  # 必须 success
/Users/wei/.n/bin/npx -y markdownlint-cli2 '**/*.md'  # 必须 0 errors
# 检查 coverage ≥ 80%(FAIL_UNDER=80)
```

**判断标准**:
- ✅ 8/8 全过 → 接受并发进程工作,进入步骤 3
- ❌ 任何 1 项失败 → 拒绝并发进程工作,git restore 撤回 + 沿原计划重做

### 步骤 3:用户授权代 commit 2 commits

**触发条件**:8/8 质量门全过 + 用户明确授权代 commit(避免擅自 commit 别人工作)。

**实施**:
```bash
# git restore 撤回 README + SESSION-STATE 修改(留给 docs closure 一起改)
git restore README.md SESSION-STATE.md

# git add feat 6 files
git add src/.../xxx.py src/.../xxx2.py tests/.../test_xxx.py ...

# commit feat
git commit -m "feat(<模块>): <主题>" -m "<改动详细>" -m "撞坑记录: 本 commit 由并发进程实施,撞坑后用户授权我代 commit"

# git add docs closure(README + SESSION-STATE + reports/<收口报告>.md)
git add README.md SESSION-STATE.md reports/<收口报告>.md

# commit docs closure
git commit -m "docs(closure): <主题>" -m "HEAD 锚 <feat hash>"

# 重写自己基于旧状态写的报告
# 然后 commit docs-only 重写报告
git add reports/<基于旧状态的报告>.md SESSION-STATE.md MODIFICATION-LOG.md README.md
git commit -m "docs(sync): <主题>" -m "撞坑记录: 本 commit 由并发进程实施,撞坑后用户授权我代 commit"
```

**commit 范本**:
- commit 1 (feat):`feat(<module>): <subject> (+/- changes / tests)`
- commit 2 (docs closure):`docs(closure): <subject>`
- commit 3 (docs-only rewrite):`docs(sync): <subject>`

**价值**:撞坑恢复 3 步范本化 = 后续撞同款问题按 3 步执行。
