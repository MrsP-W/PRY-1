### 2.2 "Load failed: 5" 实际是 false alarm 解读

| 信号 | 含义 |
|------|------|
| `Load failed: 5: I/O error` | launchctl 报错 |
| `launchctl list` 显示 job | **实际已注册** |
| `launchctl print gui/501` 显示 `type = LaunchAgent` + `path` + `program` | **plist 已 load 到 user domain** |
| `state = not running` | 预期(StartCalendarInterval 未到)|

**解释**:`launchctl load -w` 在 plist 已注册状态是 idempotent 的,但 launchctl 仍报 I/O error。这是 macOS launchctl 的已知 quirk(vs `launchctl bootstrap`),不影响实际状态。

**状态报告**:✅ **2/3 job 已 load 到 `gui/501` domain**,数字员工 plist 仍 bootout(从未 load)。

## 🆕 3. 撞坑 #91 — `docs/v0.2.67` 误判根因的真实破解

> **完整沉淀**: `memory/pitfall-91-launchd-documents-shell-operation-not-permitted.md`
> **本质**: 打破 docs/v0.2.67 §19-21 报告的撞坑 #81 误判(**TCC Python.framework 鉴权拦截**)—— 实测是 macOS `~/Documents/`(iCloud 同步目录)对 sh 脚本 **OS 层 exec 拦截**。

### 3.1 docs/v0.2.67 误判原文

> `launchctl list 显示数字员工 PID 126 / exit 126(撞坑 #81 实测命中)` — 把 exit 126 归因 **Python.framework TCC 鉴权**。

### 3.2 实测真实路径

```
bash: /Users/wei/Documents/DesktopOrganizer/我的AI员工/ops/start-digital-employee.sh: Operation not permitted
                                                                                              ^^^^^^^^^^^^^^^^^^^^^^^
                                                                                              bash exec 失败 · 不是 Python TCC
```

### 3.3 关键证据链(打破误判)

| 维度 | 实测 |
|------|------|
| `ops/start-digital-employee.sh` 文件权限 | ✅ `-rwxr-xr-x` |
| `com.apple.quarantine` 属性 | ❌ 不存在 |
| wrapper `~/bin/my-ai-employee-start` 内容 | `exec bash "/Users/wei/Documents/DesktopOrganizer/我的AI员工/ops/start-digital-employee.sh" start` |
| **关键**:`/Users/wei/Documents/DesktopOrganizer/` 在 iCloud Drive 同步范围内 | ⚠️ macOS 对 Documents 同步目录有特殊 OS 层 exec 限制 |
| stderr 前缀 | `bash: ... Operation not permitted`(**bash** exec step 失败) |
| docs/v0.2.67 误归 | TCC Python.framework 3.12 鉴权(实测**无** Python 进程启动) |

### 3.4 修正认知

- **撞坑 #81**(真存在) ≠ 数字员工 RunAtLoad=true 触发 TCC Python.framework 鉴权
- **撞坑 #91**(新发现) = macOS `~/Documents/`(iCloud 同步目录)对 `bash exec .sh` 的 OS 层限制
- **撞坑 #81 与 #91 路径叠加**:在 Documents 目录下的 `ops/start-digital-employee.sh` + RunAtLoad=true 触发的 launchd 调用,bash exec 时被 OS 层拦截,**既不**进入 Python 进程,**也**不经过 TCC(因为根本没启动 Python)
- **直接证据**:stderr 信息 `Operation not permitted`,而**不是** TCC 弹窗

### 3.5 修复路径候选(不下结论,留给 D-step)

| # | 路径 | 风险 | 推荐 |

## 🆕 4. 撞坑 #90 — launchd user-agent job 不跨 session 持久化

> **完整沉淀**: `memory/pitfall-90-launchd-domain-not-persistent.md`
> **本质**: `launchctl load -w` 注册到 user domain `gui/$UID` 的 job,**不**在 launchd 系统重启 / 用户 logout 后自动恢复,需重新 load。

### 4.1 时间线偏差

| 时间 | launchctl 状态 | 来源 |
|------|---------------|------|
| 7/8 13:13(docs/v0.2.67)| 2/3 job active(agent + imap-sync) | docs/v0.2.67 §19-21 |
| 7/8 23:0x(本次复测)| **0/3 job 注册**(全部不在 list)| 本收口 |

**9.5 小时内**:2/3 active → 0/3 active

### 4.2 根因推测

- macOS launchd **`gui/$UID` domain 不是 system-persistent**:
  - 用户 logout / system reboot → `gui/$UID` domain 整个清空
  - 需重启后重新 `launchctl load -w` 每个 plist
- **替代方案**:`launchctl bootstrap` 注册到 system target(更持久,但 user job 通常不需要)
- 用户在 docs/v0.2.67 后到本次盘点前可能经历过 logout 或 reboot(具体无法确认)

### 4.3 修复候选

- **A**:每次 reboot 后手动 `launchctl load -w × 3`(当前范本)
- **B**:加 `~/Library/LaunchAgents/` 持久化 boot script(但 `~/Library/LaunchAgents/` 本身就在 user domain)
- **C**:`sudo launchctl bootstrap` 到 system domain(高权限)
- **D**:LoginHook(`/System/Library/Login` 不可写)+ `~/.zshrc` 末加 `launchctl load -w`(低成本)

### 4.4 docs/v0.2.67 应对建议

- **不**动 v0.2.67 历史(已 commit `76cd2eb`)
- 但在 audit trail 中**显式标记**:docs/v0.2.67 §19-21 的撞坑 #81 真实根因是撞坑 #91,**不是** Python TCC
- 后续 D-step 校正 docs/v0.2.67 误判(可选 / docs-only)
