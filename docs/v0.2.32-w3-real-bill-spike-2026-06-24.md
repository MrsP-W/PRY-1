## 2. 撞坑 #49 真实暴露

### 2.1 现象

```
UnsupportedCSVVersionError: 无法嗅探支付宝账单版本: header=['导出信息：'], 已知版本 hints={2024: ('付款时间',), 2025: ('创建时间',), 2026: ('消费时间',)}
```

### 2.2 根因(3 重不一致)

| # | 不一致点 | v0.1 D7.1 faker | 真实支付宝导出 |
|---|---------|----------------|----------------|
| 1 | 前缀段 | 无 | 22 行说明文字(`导出信息:`+ 8 条特别提示 + 回单分隔行) |
| 2 | header 字段名 | 2024=`付款时间` / 2025=`创建时间` / 2026=`消费时间` | **`交易时间`**(4 个版本号无一命中) |
| 3 | `收/支` 列取值 | `支出` / `收入` | **第三种**:`不计入收支`(花呗还款/余额宝收益/提现) |

### 2.3 决策

**沿 v0.2.x spike 范本**:扩展为新版本号 `2027`(不动 2024/2025/2026 faker,向后兼容),spike 必跑。理由:
- ✅ 真实数据是 W3 真账单的**最大价值**,不解决就不算 spike
- ✅ 加 2027 parser 不破坏 2024/2025 tests(8 现有 tests 全绿)
- ✅ 沿 v0.2.18 §3 撞坑史(优先收口而非绕过)

## 5. 撞坑 #49 教训沉淀

### 5.1 faker ≠ 真实格式

D7.1 InMemory 测试用 faker 数据(基于公开文档),但支付宝**真实导出格式与文档差异**:
- header 字段名换(`交易时间` vs `消费时间`)
- 前缀段可达 22 行
- 第三种 type 值 `不计入收支`

**教训**:**faker 是契约假设,真实数据是契约证明**。真实 spike 是唯一能暴露 faker 与真实差异的手段,沿 v0.2.18 §3 撞坑 #21/#45/#46 范本。

### 5.2 spike 撞坑 3 阶段处理

| 阶段 | 动作 |
|------|------|
| 1. 暴露 | dry-run detect_version 失败 → 找到根因 |
| 2. 收口 | 扩 2027 parser(向后兼容) + 4 tests + mypy 0 |
| 3. 跑通 | 重跑 spike 验证全链路 |

**教训**:撞坑不是失败,是 spike 的**最大价值**。撞坑越多 = 真实场景覆盖越完整。

### 5.3 沿用边界严守 ✅

- ✅ `--max-rows 1`(严格只导入 1 笔)
- ✅ 5 重防误发全开(ENV + confirm + count + max-rows + DB 路径)
- ❌ 不全量导入 62 笔
- ❌ 不移动 v0.1.0 tag
- ❌ 不打 v0.2.32 tag(8/1 锚定策略)
