## 4. OutboxDispatcher × SMTPProviderFactory 接入状态复核

**v0.2.47 复核结果**(2026-06-25):

```bash
$ grep -n "SMTPProviderFactory\|provider_factory" src/my_ai_employee/scheduler/outbox_dispatcher.py
(无输出)

$ grep -n "smtp_provider\|SMTPProvider" src/my_ai_employee/policy/send_adapter.py
(无输出)
```

**结论**:**OutboxDispatcher + EmailSendAdapter 当前未接入 `SMTPProviderFactory`(v0.2.2 #8 工厂模式)**。

| 链路 | SMTPProviderFactory 接入状态 |
|------|---------------------------|
| `scripts/spike_send_100.py`(v0.2.43 解封) | ✅ **已接入**(`--smtp-provider {qq,outlook,gmail}`) |
| `EmailSendAdapter`(`policy/send_adapter.py`) | ❌ **未接入**(用 `SmtpLibTransport` / `InMemorySmtpTransport`) |
| `OutboxDispatcher`(`scheduler/outbox_dispatcher.py`) | ❌ **未接入**(通过 `EmailSendAdapter` 间接持有 transport) |
| `OutboxEntry.source` 字段 | `outbox:<source>:<run_id>` 用 `'qq'/'outlook'/'gmail'` 但**未路由到对应 provider** |

**修复方向**(用户授权后可执行):
1. `EmailSendAdapter.__init__` 增加 `smtp_provider: Literal["qq","outlook","gmail"]` 参数
2. 内部通过 `SMTPProviderFactory.create(provider)` 拿对应 transport(沿 v0.2.2 #8 工厂模式)
3. `OutboxEntry.source` 字段决定 `smtp_provider`(auto-routing)
4. **风险**:改 `EmailSendAdapter` 签名 = 多个 test fixtures 需同步 + 撞坑 #18 风险门控

**建议**:**不阻塞 8/1 tag 决策**(v0.2.2 #8 已实化工厂类,未接入 = B 类延后项,延后到 8/1 之后单独 PR)。
