## 1. 修复目标

**用户原话**:"B(L2 fingerprint sign-lock 小修,解决 v0.2.27 暴露的 267 对偶然跨源 L2 命中风险)"

**目标**:`normalize_fingerprint` 升级为支持 `sign` 参数,消除 sign 错乱的偶然跨源命中。

## 2. 设计决策

### 2.1 三方案对比

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| A. 直接用 sign 拼 canonical | `f"{date}|{sign}|{amt}"` | 改动小 | sign 拼前面要严判 |
| B. 用有符号 Decimal 字符串 | `f"{date}|{signed_amt:.2f}"` | 语义清晰 | 默认 abs() 路径不保留 |
| **C. 加可选 sign 参数(默认 None 向后兼容)** | `normalize_fingerprint(..., *, sign: int \| None = None)` | **D6.2 + D7.2 已有测试零破坏** + 显式启用 | 双路径维护 |

### 2.2 决策:方案 C

**理由**:
- ✅ 向后兼容:D6.2 + D7.2 + D6.6 已有测试用 `sign=None`(默认)继续走 abs() 路径
- ✅ 显式启用:transaction_adapter.py:192 显式传 `sign=+1/-1`,消除偶然跨源
- ✅ 严判边界:`type(sign) is int` + `sign in (+1, -1)` 严判(沿工厂层 type() is int 范本)
- ✅ sign 与 amount 符号独立:业务侧 RawTransaction.amount 来自 parser 可能有 ±,而 sign 由 transaction_adapter 从 raw.type(已归一化)派生,二者独立

## 3. 实施细节

### 3.1 核心改动:fingerprint.py

```python
def normalize_fingerprint(
    date: date | str,
    amount: Decimal | int | float | str,
    counterparty: str,
    *,
    sign: int | None = None,  # ← v0.2.28 新增
) -> str:
    date_norm = _normalize_date_value(date)
    amount_norm = _normalize_amount_value_with_sign(amount, sign=sign)  # ← 升级
    counterparty_norm = _normalize_counterparty_value(counterparty)
    canonical = _canonical_payload(date_norm, amount_norm, counterparty_norm)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()[:_FINGERPRINT_LENGTH]


def _normalize_amount_value_with_sign(
    value: Decimal | int | float | str,
    *,
    sign: int | None,
) -> str:
    # sign 严判(沿工厂层 type() is int 范本)
    if sign is not None:
        if type(sign) is not int:  # noqa: E721
            raise TypeError(f"sign 必须是 int 或 None,实际 type={type(sign).__name__}, value={sign!r}")
        if sign not in (+1, -1):
            raise ValueError(f"sign 必须是 None / +1 / -1 之一,实际 sign={sign!r}")

    # ... amount 类型严判沿 _normalize_amount_value ...

    amt = Decimal(str(value)).quantize(_AMOUNT_QUANT, rounding=ROUND_HALF_UP)

    if sign is None:
        return f"{abs(amt):.2f}"  # 向后兼容 D6.2/D7.2
    return f"+{abs(amt):.2f}" if sign == +1 else f"-{abs(amt):.2f}"
    # sign=+1/-1:统一用 abs(amt) 防 amount 自身符号干扰,sign 由调用方锁定
```

### 3.2 业务侧启用:transaction_adapter.py:192

```python
# v0.2.28 L2 fingerprint sign-lock:消除 v0.2.27 暴露的 267 对偶然跨源 L2 命中

# type="支出"(微信付/支付宝支)→ sign=+1;type="收入"(微信收/支付宝收)→ sign=-1
```

## 5. 撞坑观察与教训

### 5.1 撞坑 #42(本轮新增)— sign 与 amount 方向矛盾过度严判

**现象**:Case 5 最初设计为"sign=+1 但 amount=-38.50 → ValueError"(防止 sign 与 amount 矛盾)。

**根因**:业务侧 RawTransaction.amount 来自 parser 可能有 ±,而 sign 由 transaction_adapter 从 raw.type(已归一化)派生,**二者独立**(amount 符号不代表业务方向,只有 type 代表)。

**修法**:删除矛盾严判,改为"sign=+1 统一返回 +abs(amt)"(无论 amount 自身符号)。Case 5 升级为 `test_fingerprint_sign_lock_amount_sign_independent`。

**教训**:严判边界要贴合业务语义,不要造"看起来合理但实际不必要"的硬约束。沿 [[撞坑 #27]] — 入口严判 vs 业务约束分离。

### 5.2 撞坑 #43(本轮新增)— 现有测试 case 与新业务行为对齐

**现象**:13 个测试 case 失败,根因是现有 case 用 `sign=None`(默认 abs)插入已有交易,但 transaction_adapter.py:192 升级后用 `sign=+1`,二者**指纹不匹配**。

**修法**:5 处现有 case 升级 `sign=+1` 与业务侧对齐(e2e test_s6_cross_source_dedup 期望值改用 sign 派生 wechat_pos_fps)。

**教训**:**测试是业务契约的体现**,业务逻辑升级时测试必须同步升级,否则会出现"业务对、测试错"或"业务错、测试对"的双向误解。

### 5.3 撞坑 #44(本轮新增)— ruff F841 隐藏修复 + noqa 注释

**现象**:`wechat_fps` / `alipay_fps` 变量计算后未直接使用(改用 sign 派生的 `wechat_pos_fps` / `alipay_pos_fps`),触发 ruff F841。

**修法**:
- 方案 A:`del wechat_fps_unused` + `# noqa: F841` — 保留计算表达计算过程(沿 v0.2.x debug helper 范本)
- 方案 B:完全删除(更彻底)

**决策**:选方案 A 保留计算表达,因为未来回看时能直接看到"业务层曾经计算过全集 fp,后续才决定用 sign 过滤"。
