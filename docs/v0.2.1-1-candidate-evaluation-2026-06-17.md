### 2.1 方案 A 工厂模式(沿 [[d6-wechat-bill-launch]] 范本)

```python
class SMTPProvider(Protocol):
    def send(self, message: EmailMessage) -> SendResult: ...
    def healthcheck(self) -> bool: ...

class QQProvider: ...          # 已实化(沿 D5.6.5)
class OutlookProvider: ...    # 待实化(smtp.office365.com:587 + App Password)
class GmailProvider: ...      # 待实化(smtp.gmail.com:587 + OAuth 2.0)

class SMTPFactory:
    @staticmethod
    def create(provider_name: str, credentials: Keychain) -> SMTPProvider:
        if provider_name == "qq": return QQProvider(credentials)
        if provider_name == "outlook": return OutlookProvider(credentials)
        if provider_name == "gmail": return GmailProvider(credentials)
        raise UnknownProviderError(provider_name)
```

## 3. outlook SMTP 部分解封路径(用户授权后)

### 3.1 outlook 不依赖 OAuth 的原因

- Microsoft 365 SMTP 鉴权: **Basic Auth(已 deprecated) / OAuth 2.0 / App Password**
- App Password 路径: 用户在 Microsoft 账户设置里创建 App Password(不需要 OAuth 库)
- 沿 D5.6.5 QQProvider 范本 + 新增 `OutlookProvider(smtp.office365.com:587 + App Password)`

### 3.2 部分解封 4 commits(仅 outlook,不包含 Gmail)

**注意**:GmailProvider 跳过(等 #6 OAuth 抽象层落地后单独 commit)

### 3.3 outlook SMTP 配置要求

```

smtp_port = 587
username = <user_email>@outlook.com
password = <App Password>(16 字符,Microsoft 账户安全设置生成)
encryption = STARTTLS(端口 587)/ SSL(端口 465)
```

## 4. Gmail SMTP 完全阻塞分析

### 4.1 阻塞原因

Gmail 自 2024 年起**强制要求 OAuth 2.0** 进行 SMTP 认证,不再支持"Less secure apps"或"App Password"路径(Google Workspace 管理员可在某些场景下启用,但个人 Gmail 账户完全 OAuth 化)。

### 4.2 OAuth 2.0 抽象层要求(沿 v0.2.1 #6)

```python
class OAuth2Provider(Protocol):
    def get_auth_url(self) -> str: ...
    def exchange_code(self, code: str) -> OAuth2Token: ...
    def refresh_token(self, token: OAuth2Token) -> OAuth2Token: ...

class MicrosoftOAuth2: ...    # msal
class GoogleOAuth2: ...       # google-auth

# Keychain 4 字段扩展
{
    "provider": "gmail",
    "access_token": "...",
    "refresh_token": "...",
    "expires_at_ms": 1234567890000,
}
```
