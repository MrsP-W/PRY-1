# v0.2.53.26 HTML dry-run 三门结果展示联调(2026-06-26)

> **阶段**:v0.2.53.26 = HTML 升级到三门联调展示
> **基线**:沿 v0.2.53.25 docs-only 收口 / commit `b490f6a` = 2496 passed / 1 skipped / 88.76% / 116 files / 164 MD

## 1. 本轮目标

v0.2.53.22 决策矩阵路径 3.5(`DASHBOARD_WRITE_API=1` + `BUSINESS_WRITER_ENABLED=1` + confirm_text 全部通过)已经能在 backend 端产出 `approval_gate_passed=True` 的 dry-run 响应(200 OK)。

但 Dashboard HTML 端(v0.2.53.11 引入的 dry-run 面板)只能展示**单一状态**:从 `approval_gate_passed` 字段反向推断。用户没法直观看到当前 3 道门各自的开关 + 综合状态属于哪一类。

本轮目标:**让 HTML 前端可视化 3 道门 + 4 类 outcome**,把 v0.2.53.22 决策矩阵的 8 路径收敛到 4 类状态展示,前端只读不写。

## 2. 实现路径(2 步,2 文件,2 commit)

### Step 1: backend `responses.py` 暴露 4 字段 + helper(commit `b490f6a`)

- 新增字段 `approval_gates.business_writer_enabled`(沿 v0.2.53.22 `writer_enabled` 暴露)
- 新增字段 `approval_gates.v0_2_53_26_dry_run_status`(嵌套 4 字段 dict)
- 新增 helper `_dry_run_three_gate_status(approval_gate) → dict[str, str]`:
  - `first_gate` = "open"/"closed"(对应 `DASHBOARD_WRITE_API` env)
  - `second_gate` = "confirm_required_per_action"(confirm_text 为 POST 请求级字段,非全局 env 状态)
  - `third_gate` = "open"/"closed"(对应 `BUSINESS_WRITER_ENABLED` env)
  - `outcome` ∈ {"disabled", "writer_required", "dry_run_ready"}(3 态,不引入 "needs_confirmation",因为 confirm_text 是 POST 阶段硬约束,非 GET /api/status 范围)

### Step 2: frontend HTML hydrate 3 门 + 4 outcome 颜色(docs commit)

- 顶部 status strip 新增 `<span class="pill">Dry-run 检测中</span>` + dot
- 升级 `renderStatus()` 提取 `writeApi` / `writerEnabled` / `dryRunStatus`
- 新增 `renderDryRunThreeGate()` helper + 4 outcome 颜色映射:
  - `disabled` → red("双门都关 — 默认只读")
  - `writer_required` → yellow("Write API 已开,但 BusinessWriter 未启用")
  - `dry_run_ready` → green("三门全开 — 路径 3.5 dry_run=True 将返回 200 OK")
- system 视图新增 card "审批门(三门联调 · v0.2.53.26)",展示 4 项:
  - 第一道门:`DASHBOARD_WRITE_API` · open/closed
  - 第二道门:`confirm_text` 链路 · confirm_required_per_action(POST 时校验)
  - 第三道门:`BUSINESS_WRITER_ENABLED` · open/closed
  - Dry-run 当前结果:outcome + 描述
