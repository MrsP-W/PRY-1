## 2. 撞坑 #61 详细分析(本轮新增)

### 2.1 现象

v0.2.51 烟测 `EmailSendAdapter(source="outlook", smtp_provider="outlook")` 跑 `send_and_emit` 时报:

```
SMTPConnector / False False False / TypeError: connect() got unexpected keyword argument 'timeout'
```

### 2.2 根因

`SMTPProviderFactory.create()` 返回 `SMTPConnector`(`src/my_ai_employee/connectors/smtp.py:464`),而 `EmailSendAdapter._smtp_transport` 期望 `_SMTPTransportLike` Protocol(`src/my_ai_employee/policy/send_adapter.py:89`)。两者**协议语义不匹配**:

| 类 | 关键方法签名 | 角色 |
|---|------------|------|
| `SMTPConnector`(高层) | `async def connect(self) -> None`(551 行,无 host/port/timeout)| 业务封装,自管理 Keychain 读 + 内部 transport 注入 |
| `SMTPTransport` Protocol | `def connect(self, host, port, *, timeout=30.0) -> None`(smtp.py:98)| 底层 transport 抽象,被 EmailSendAdapter 直接调用 |

v0.2.51 误把 `SMTPConnector`(高层)赋给 `_smtp_transport`,后续 `transport.connect(host, port, timeout=30.0)` 落到 `SMTPConnector.connect()`,签名不匹配抛 TypeError。

### 2.3 修复

`src/my_ai_employee/policy/send_adapter.py:657-696` 改为:

```python
# 关键修复:取底层 SMTPTransport(SMTPConnector.transport 属性),
# 而非把 connector 塞进 _smtp_transport(协议不匹配)
connector = SMTPProviderFactory.create(provider=smtp_provider, email=placeholder_email)
underlying_transport = connector.transport
if underlying_transport is None:
    # SMTPProviderFactory.create() 默认会注入 SmtpLibTransport(),但 D5.1-fix
    # 修订后允许 transport=None 创建(防"假成功");此处显式 fallback 兜底
    underlying_transport = SmtpLibTransport()
self._smtp_transport = underlying_transport
# 暴露 provider 默认配置(供 OutboxDispatcher / send_and_emit 参考)
self._smtp_provider = smtp_provider
self._provider_default_host = connector.server_host
self._provider_default_port = int(connector.server_port)
self._provider_default_email = placeholder_email
```

**关键约束**:
- `smtp_provider` 与 `smtp_transport` 维持**互斥严判**(同传 ValueError)
- `connector.transport` 为 None 时显式 `SmtpLibTransport()` fallback(防 D5.1-fix "假成功" 警告场景)
- 真实发件邮箱来源 = `entry.sender_email`(由 OutboxDispatcher 在 v0.2.51.1 注入),SMTPProviderFactory 的 email 字段仅作占位

### 2.4 测试覆盖(2 新 tests)

| 测试 | 验证 |
|------|------|
| `test_smtp_provider_mode_assigns_underlying_transport_not_connector` | ① `_smtp_transport` 是 `SmtpLibTransport` 实例(非 `SMTPConnector`)② 协议方法齐全 `connect/login/send_message/quit`(用 `hasattr` 校验,因 `SMTPTransport` 是 Protocol 不能 `isinstance`)③ provider 默认配置字段正确填充(`_smtp_provider=outlook` / `_provider_default_host=smtp.office365.com` / `_provider_default_port=465` / `_provider_default_email=outlook@my-ai-employee.local`)④ 协议签名校验: `connect(host, port, timeout=0.001)` 不抛 `TypeError`(`SmtpTransportError`/`OSError`/`TimeoutError`/`smtplib.SMTPException` 都算预期)|
| `test_send_and_emit_smtp_provider_factory_end_to_end` | 端到端:`monkeypatch` `SMTPProviderFactory.create` 注入 `InMemorySmtpTransport` → `EmailSendAdapter(smtp_provider="outlook")` → `send_and_emit` 走完整 4 步 `connect/login/send_message/quit` 路径 → `outbox status: APPROVED → SENT` 全绿 |

## 3. 撞坑 #62 详细分析(本轮新增)

### 3.1 现象

`make alembic` 失败时仍报 exit 0(head 把 alembic 退出码吞了)。

### 3.2 根因

`Makefile:240` 原版:
```makefile
@$(PYTHON) -m alembic upgrade head --sql 2>&1 | head -50
```

`head` 是 pipe 末位命令,`$?` 在 pipe 上下文里指 `head` 的退出码(0),alembic 的失败被掩盖。

### 3.3 修复

`Makefile:240` 改为:
```makefile
@echo "$(BLUE)🗄️  alembic upgrade head --sql(... v0.2.52 P2 修:先写临时文件再 head 避免 head 吃退出码)$(RESET)"
@$(PYTHON) -m alembic upgrade head --sql > /tmp/alembic_head.sql 2>&1 ; status=$$? ; head -50 /tmp/alembic_head.sql ; exit $$status
```

**关键技术点**:
- `> /tmp/alembic_head.sql 2>&1` 把 alembic 输出写临时文件
- `; status=$?` 用分号分隔命令避免 pipe,`$?` 直接捕获 alembic 退出码
- `exit $status` 透传给 `make`(沿 D3.3.3 范本)

**注意 zsh 坑**:`status` 是 zsh 只读变量,改用别的名字(`rc`/`alembic_status`)或在 `bash` 子 shell 里跑。

### 3.4 验证

```bash
$ cd 我的AI员工 && make alembic > /dev/null 2>&1; echo "exit=$?"
exit=0
```
