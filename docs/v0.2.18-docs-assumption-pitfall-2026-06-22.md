## §2 docs 假设错误类撞坑定义

### 2.1 类别边界

**docs 假设错误类**:文档凭印象/记忆/假设推断,而非实测/find/ls 验证所导致的撞坑。

### 2.2 与撞坑史 5 类的区别

| 撞坑史 | 触发源头 | 影响范围 |
|--------|----------|----------|
| **撞坑史 5 类**(基线)| pwd 漂移 / 路径错位 / 边界误判 / SIGKILL 137 误报 / 连写错误 | 工具/路径/命令/环境 |
| **撞坑史 6 类 🆕**(本次新增)| **docs 凭印象假设,实测不符** | **信息源头/文档准确性** |

### 2.3 触发场景模式

```
docs 写文档(凭印象)  →  实测验证(撞坑)  →  降级应对  →  docs 更新(范本沉淀)
   ↑                       ↓
   └─── 撞坑史 6 类沉淀 ──┘
```

**核心要点**:docs 写错 → 实测暴露 → 范本沉淀,**docs 错误不是终点,而是撞坑史扩类的契机**。

### 2.4 撞坑史 6 类完整清单(本次专项固化)

| # | 类别 | 代表撞坑 | 触发源头 | 沉淀范本 |
|---|------|----------|----------|----------|
| 1 | pwd 漂移 | 撞坑 #21 | Bash 跨项目 pwd 漂移 | `cd <项目路径>` 显式修正 |
| 2 | 路径错位 | 撞坑 #22 | `src/myaiemployee` 实际 `my_ai_employee` | `ls src/` 验证 |
| 3 | 边界误判 | 撞坑 #23 | 假设数量/层级/状态不符 | `find` / `ls` 验证 |
| 4 | SIGKILL 137 误报类 | 撞坑 #25 | `.venv/bin/alembic` framework 签名失效 | `uv run` 包装器绕开 |
| 5 | 连写错误 | 撞坑 #26 | 下划线 vs 无下划线 | `ls src/` 验证 |
| **6 🆕** | **docs 假设错误** | **撞坑 #24 / #27 / #28** | **docs 凭印象,实测不符** | **`find` / `ls` 实测回填** |

## §4 docs 假设错误类撞坑 3 步实战范本(撞坑恢复 3 步实战演练 11)

> **沿用范本**:[[v0.2.17-6-23-readiness-final-sprint-2026-06-22]] §3 撞坑沉淀 + [[v0.2.15]] §3 撞坑恢复 3 步实战演练 9 范本
> **本次升级**:从"通用 3 步"细化到"docs 假设错误类专项 3 步"

### Step 1:docs → 实测差距识别

**目标**:识别 docs 假设与实测数据的偏差。

**动作清单**:

1. 提取 docs 中的假设(数量/路径/层级/状态)
2. 选用验证命令(按撞坑类型选 `ls` / `find` / `grep` / `wc -l`)
3. 实测并对比差异(实测数据 vs docs 假设)

**踩坑警示**:

- ❌ 不要"先行动后验证" → 必须 docs → 实测 → 行动
- ❌ 不要"凭印象验证" → 必须用工具实测

### Step 2:降级应对 + 实测数据回填

**目标**:用实测数据降级应对,不让 docs 错误阻塞整体流程。

**动作清单**:

1. 选用降级策略(沿用撞坑 #21-#28 范本)
2. 实测数据回填到 docs(标注"实测"而非"假设")
3. 标注影响范围(是否阻塞 6/23 实操启动)

**踩坑警示**:

- ❌ 不要"硬撞 docs 假设" → 必须实测优先,docs 跟随
- ❌ 不要"降级但不回填" → docs 必须标注实测数据

### Step 3:docs 更新 + 范本沉淀

**目标**:把 docs 错误转化为撞坑史扩类契机。

**动作清单**:

1. 更新 docs(数量/路径/层级全部用实测数据)
2. 沉淀范本(沿用 §3 撞坑 #24-#29 范本格式)
3. 检查员 audit 标注"docs 假设校验"环节(本次 v0.2.18 §6)

**踩坑警示**:

- ❌ 不要"修了不沉淀" → 必须留下范本供后续撞坑参考
- ❌ 不要"个人范本" → 必须沿用前序版本范本格式
