## 三、4 类实写流程串联(handler + audit + HTML 端到端)

### 3.1 `outbox.approve` 端到端

```
[handler]
1. evaluate_writer_dry_run(env=POST body, ctx=DashboardContext)
   - DASHBOARD_WRITE_API ✓ + confirm_text ✓ + BUSINESS_WRITER_ENABLED ✓
   - ctx.business_writer is BusinessWriterImpl ✓
2. writer.approve_outbox(target_id, audit)
   - OutboxStore.update_status(id, APPROVED, from_status=PENDING_SEND, last_approved_at_ms=now_ms)
   - 异常 = OutboxIllegalTransitionError 透传
3. _audit_log(action='outbox.approve', target_id, write_executed=True, affected_id, executed_at_ms)
4. WriteResult(success=True, write_executed=True, affected_id, audit_id)

[HTML]
1. fetch('/api/approval-gate/actions', POST {action: 'outbox.approve', target_id, confirm_text: 'CONFIRM_WRITE'})
2. inspector 渲染:
   - HTTP 200:「✅ 已审批(outbox {affected_id}, audit #{audit_id}, {executed_at})」
   - HTTP 409:「❌ 写入失败 illegal_transition(状态机不允许)」
   - HTTP 500:「❌ 写入失败 internal_error(OutboxStore 异常)」
3. API 离线 → 「⏸️ API 离线 · 启动 `make dashboard-api`」(write_executed=false)
```

### 3.2 `outbox.cancel` 端到端(同 3.1 · from_status 允许 PENDING_SEND + APPROVED)

### 3.3 `notes.confirm` 端到端

```
[handler]
1. writer.confirm_note(target_id, audit)
   - NoteConfirmServiceImpl.confirm_note → NoteStore.mark_archived
2. _audit_log(action='notes.confirm', target_id, write_executed=True, affected_id=target_id)
3. WriteResult(success=True, write_executed=True, affected_id=target_id, audit_id)

[HTML] 同 3.1
```

### 3.4 `finance.dismiss_anomaly` 端到端

```
[handler]
1. writer.dismiss_anomaly(target_id, audit)
   - AnomalyDismissalServiceImpl.dismiss(anomaly_id, reason=audit.reason)
   - 返回 DismissalResult(success, anomaly_id, error, reason)
2. _audit_log(action='finance.dismiss_anomaly', target_id, write_executed=success, affected_id, error)
3. WriteResult(success, affected_id, error, audit_id)

[HTML]
1. inspector 渲染:
   - success=True:「✅ 已忽略({anomaly_id}, audit #{audit_id})」
   - success=False:「❌ 忽略失败({error})」(注意:dismissal 失败 write_executed=False,与 outbox 不同)
```

**关键差异**:
- outbox / notes = 实写动作(失败也落档,write_executed=True)
- anomaly dismissal = 业务校验可拒(失败不落档 write_executed=False? 沿 v0.2.53.20 沿用范本)

## 四、audit_log 落档时机与失败语义

### 4.1 落档时机(沿撞坑 #65 + v0.2.53.20 §3)

| 业务结果 | write_executed | audit 落档 | error 字段 |
|----------|----------------|------------|------------|
| 实际写入成功 | `true` | ✅ | `null` |
| 状态机不允许(outbox.approve SENT→APPROVED) | `true` | ✅ | `illegal_transition` |
| writer 内部异常 | `true` | ✅ | `internal_error` + reason 透传 |
| anomaly dismiss 失败 | `false` | ❌ | `dismiss_failed` 走 dry-run 路径 |
| 路径 1-3(未达路径 4) | `false` | ❌ | — |

### 4.2 关键不变式

- ✅ 路径 4 触发的实写,**任何结果都落档**(成功 + 异常),审计不可绕过
- ✅ `error` 字段填具体原因(illegal_transition / internal_error / invalid_target_id)
- ✅ `write_executed` 与 `error` 字段**可同时存在**(成功:error=null;失败:error=原因)
- ❌ 路径 1-3 不落档(沿 v0.2.53.11 范本:未达实写不污染审计表)
