## 1. 微信 App 导出步骤(8 步)

### 1.1 入口路径

```
微信 App → 我 → 服务 → 钱包 → 账单 → 右上角 ··· → 账单下载
```

**界面截图位**（待补真实截图）：

| 步骤 | 路径 | 关键点 |
|------|------|--------|
| 1 | 微信 App 首页 → **我** | 底部 Tab 第 4 个 |
| 2 | **服务** | "我"页面 → 顶部"服务"入口（部分版本为"支付"） |
| 3 | **钱包** | 服务页 → 金融理财区 → 钱包 |
| 4 | **账单** | 钱包页 → 顶部"账单"按钮 |
| 5 | 右上角 **···** | 账单页右上角三点菜单 |
| 6 | **账单下载** | 弹出菜单 → "账单下载" |
| 7 | 选择时间范围 | 支持近 1 个月 / 3 个月 / 自定义 |
| 8 | 邮箱接收 | 填写接收邮箱（QQ 邮箱 / 163 / Gmail 等） |

### 1.2 时间范围与版本对应

微信账单 CSV 格式**按时间范围**有差异（D6.1 范本）：

| 时间范围 | Parser | 字段特征 | 备注 |
|----------|--------|---------|------|
| **2024 年及之前** | `WeChatCSV2024Parser` | 列名含 `交易时间` / `交易类型` / `金额(元)` | 公开文档稳定 |
| **2025 年** | `WeChatCSV2025Parser` | 列名含 `日期` / `收/付` / `金额` | 2025 改版后字段 |
| **2026 年** | `WeChatCSV2026Parser` | **待用户真实样本** | D6 留 `NotImplementedError` 占位 |

**InMemory faker 合成(D6.1)**：
- 当前 D6.1 已用 faker 合成 3 版本各 5 笔样本来覆盖 `detect_version` 嗅探
- W1 D6.5 spike 100 笔用同一套 faker 样本

## 2. CSV 文件处理

### 2.1 邮箱接收 + 解压

1. **查收邮件**：微信支付会发一封标题为"微信支付账单"的邮件
2. **下载附件**：附件为 `.zip` 压缩包（**不是直接 CSV**）
3. **解压**：得到 `微信支付账单(YYYYMMDD-YYYYMMDD).csv` 文件
4. **检查编码**：UTF-8 with BOM（Excel 双击打开可能乱码，见 §2.3）

### 2.2 文件命名规范

解压后的文件名格式：

```
微信支付账单(20260101-20260614).csv
微信支付账单(20250601-20260531).csv
微信支付账单(20240601-20240531).csv
```

**D6.1 `detect_version` 嗅探策略**：
- 读 CSV 前 5 行 header 列名
- `交易时间` + `交易类型` → 2024 parser
- `日期` + `收/付` → 2025 parser
- 其他 → 2026 parser（**当前 raise NotImplementedError**）

### 2.3 Excel 打开乱码解决方案

**问题**：Excel 双击 CSV 默认按 GBK 解码 → 中文乱码

**解决方案**（**任选一种**）：

| # | 方案 | 操作 |
|---|------|------|
| 1 | **Numbers 打开** | macOS 自带 Numbers 直接拖入 CSV → 正常显示 |
| 2 | **Excel 数据导入** | Excel → 数据 → 从文本/CSV → 选 UTF-8 编码 |
| 3 | **VSCode / 文本编辑器** | 直接看原始 UTF-8 内容 |
| 4 | **命令行 iconv 转码** | `iconv -f UTF-8 -t GBK input.csv > output_gbk.csv` |

**CLI 导入不需要打开 Excel**，直接传路径给 `scripts/import_wechat.py` 即可。

## 3. CLI 导入流程

### 3.1 默认 InMemory 模式(开发 + D6.5 spike)

```bash
# 跑 faker 合成 3 版本各 100 笔 InMemory(默认,D6.5 spike)
.venv/bin/python scripts/import_wechat.py \
  --csv-path tests/fixtures/wechat_faker/2026_sample.csv

# 输出示例(实跑结果,2026-06-14)
# parsed=5 inserted=5 categorized=5 duplicates=0 needs_confirm=0 failed=0
```

**不传 `WECHAT_REAL_IMPORT=1` env 变量时**：
- 默认走 InMemory / faker 合成路径
- 真实 CSV 不写入 transactions 表
- 仅做格式验证 + detect_version 嗅探

### 3.2 真实导入模式(W2 6/23,4 重防误发)

**前置条件**：用户已从微信 App 导出 CSV 并解压

**4 重防误发(沿 D5.6.5 范本)**：

```bash
# 1) 必须设 env 门控
export WECHAT_REAL_IMPORT=1

# 2) --csv-path 必传(沿用 D5.6.5 --recipient 范本)
# 3) --max-rows 1 限制单次导入行数(防误传大文件)
# 4) --confirm 必传确认文本(防脚本误触发)
# 5) --count 1 限制总批次数
.venv/bin/python scripts/import_wechat.py \
  --csv-path /tmp/real_wechat_2026.csv \
  --max-rows 1 \
  --confirm "yes-i-understand-this-imports-real-bill" \
  --count 1
```

**4 重防误发 + 1 env 门控全过条件**：
- ✅ `WECHAT_REAL_IMPORT=1` env 变量已设
- ✅ `--csv-path` 指向真实 `.csv` 文件(非 faker 样本)
- ✅ `--max-rows 1` 限制单次最多 1 行
- ✅ `--confirm` 文本**完全匹配** "yes-i-understand-this-imports-real-bill"
- ✅ `--count 1` 限制总批次数 1 次

**任一条件不满足 → CLI 拒绝并退出码 1**。

### 3.3 退出码约定(沿 D5.6.5 范本)

| 退出码 | 含义 |
|--------|------|
| 0 | 成功(全部行 parsed + inserted) |
| 1 | 4 重防误发未全过 / 入参非法 |
| 2 | 部分行 failed(>0 行抛 OperationalError 等技术失败) |
| 3 | DB 锁 / 连接失败(透传 OperationalError) |

## 4. 3 层去重模型(D6.2 + D6.4 契约)

### 4.1 L1 源内幂等(硬约束,UNIQUE 业务阻断)

**触发条件**：
```python
UNIQUE(source, external_transaction_id)  # D6.4 0007 migration 必加
```

**行为**：
- INSERT 时 `IntegrityError` → `TransactionDuplicateError` → Adapter 接住 → 写 events(`event=TRANSACTION_BLOCKED`)
- 不**走技术失败入口**(OperationalError 等才走技术失败)
- `duplicates` 计数 +1,继续下一行

**用户感知**：CLI 输出 `duplicates=N`(N = 源内重复行数),**不报错**(业务阻断≠系统错误)。

### 4.2 L2 跨源候选(软标记,fingerprint INDEX)

**触发条件**：
```python
INDEX(normalized_fingerprint)  # 32 chars SHA-256 hex
```

**跨源候选查找逻辑**(沿 D6.5 adapter 实现)：
```python
candidates = [
    candidate
    for candidate in store.find_candidates_by_fingerprint(fingerprint)
    if candidate.source != source  # 跨源过滤(D6='wechat',D7='alipay')
]
```

**行为**：
- `len(candidates) == 0` → 直接插入,`needs_confirm=False`
- `len(candidates) >= 1` → 插入但设 `needs_confirm=True` + `candidate_match_id=candidates[0].id`
- 状态机推到 `NEEDS_CONFIRM`,等待用户 1-click 确认

**W2 真实 1 笔时不会触发 L2**(D6 阶段无 D7 支付宝数据,只有 D6='wechat' 单源,L2 同源不命中)。

### 4.3 L3 模糊匹配(只标记,绝不 delete/update)

**核心契约**(plan §5 防误合并 5 重点)：

| # | 防护点 | 实现 |
|---|--------|------|
| 1 | **绝不**自动 delete/update 候选行 | L3 只设 `needs_confirm` + `candidate_match_id` |
| 2 | 等用户 1-click 确认 | 状态机推到 `NEEDS_CONFIRM`,人工走 CLI `--confirm-match` |
| 3 | 防日期相邻 ±1 天误命中 | 指纹算法只取日期 `YYYY-MM-DD` |
| 4 | 防商家名轻微变化误命中 | `re.sub(r"\*+", "", x).strip().lower()` 归一化 |
| 5 | 防退款抵消误命中 | 退款作独立行(负数金额),L1 不会命中,L2 也不会同源命中 |

## 5. 常见问题 FAQ

### Q1:为什么我的 CSV 在 Excel 打开是乱码？

A: Excel 默认按 GBK 解码 UTF-8 with BOM 文件。**用 Numbers 打开** 或 **用 Excel 数据导入向导指定 UTF-8**(见 §2.3)。

### Q2:为什么导入后 `duplicates > 0` 但没报错？

A: 这是**业务阻断**(L1 源内幂等)的正常行为。表示该 `(source, external_transaction_id)` 已存在(可能是重复导入)。**CLI 退出码 0**(整体成功)。

### Q3:为什么 `needs_confirm > 0`？

A: 表示有跨源候选(D6='wechat' 已有同 fingerprint 交易)。**当前 D6 阶段无 D7 支付宝数据**,W1 D6.5 InMemory 测试时 `needs_confirm=0` 是预期。

W2 真实 spike 时如果出现 `needs_confirm > 0`,表示有**重复交易**(可能用户多次导出),需要用户 1-click 确认或忽略。

### Q4:为什么 2026 版本 CSV 导入报 NotImplementedError？

A: D6.1 阶段 2026 parser 字段**待用户真实样本**确认。当前的 faker 合成 2026 样本是按公开文档猜字段,W2 6/23 真实 spike 时如果报这个错,请把 CSV 样例前 5 行 header 贴给我,我会修正 `WeChatCSV2026Parser` 字段映射。

### Q5:为什么 `failed > 0` 退出码是 2？

A: `failed` 是技术失败计数(OperationalError / DB 锁 / 接口错误)。退出码 2 表示**部分行成功 + 部分行失败**,需查看 events 表 `event=TRANSACTION_FAILURE` 详情。

### Q6:W2 真实 1 笔 spike 前我能本地测试吗？

A: 可以。InMemory 模式不需真实 CSV,直接 `make test` 跑 D6.5 专项(`pytest tests/core/test_transaction_adapter.py -v`)3/3 全过即表示契约就绪。
