## 1. 背景与目标

**v0.2.53.28 已完成** `business_writer_ready = env AND impl` 第三道门真实状态判定(见 [docs/v0.2.53.27-business-writer-opt-in-2026-06-26.md](v0.2.53.27-business-writer-opt-in-2026-06-26.md)),但 status payload 只暴露合并字段 `business_writer_ready` 一个布尔值,前端 inspector 无法区分"env 已开但 Impl 未注入(env_only marker)"vs"env 未开(default disabled)"。

**v0.2.53.29 目标**:

1. **payload 暴露 3 字段**:让前端 inspector 看到 env / impl / ready 三态独立状态,而非合并 ready 单字段。
2. **POST 路径 501 文案边界化**:当 env 已开但 Impl 未注入(因 `DASHBOARD_REAL_DB` 未开 / `session_factory` 失败 / `BusinessWriterImpl` 构造失败),响应 `reason` 明确告知"需 `DASHBOARD_REAL_DB=1` + session 成功 + Impl 构造成功",而非默认 "BusinessWriter 未启用"文案(会让用户误以为是 env 未开)。
3. **前端 3 badge 渲染**:HTML inspector 单 `Writer env` badge 升级为 3 badge(`Writer env` / `Writer Impl` / `Writer ready`),逐一展示 + 颜色 tone 与后端 3 字段对齐。

### 2.1 payload 暴露字段

ApprovalGate 决策 payload(无论 status code)新增 3 字段:


字段含义:

| 字段 | 含义 | 数据源 |
|------|------|--------|
| `business_writer_env_enabled` | `BUSINESS_WRITER_ENABLED` env 是否设为 truthy 字面量 | `is_business_writer_enabled()`(env 直接读) |
| `business_writer_impl_injected` | `BusinessWriterImpl` 是否实际注入到 ctx(handler 透传 ctx 的 `is_business_writer_impl_injected()`)| `DashboardContext.is_business_writer_impl_injected()` |
| `business_writer_ready` | env AND impl 双开(合并字段,前端红色 "no" 提示) | `bool(env) AND bool(impl)` |

## 4. 撞坑沉淀(撞坑 #68 = v0.2.53.29 三字段对齐范本)

### 撞坑 #68(本轮新增)· 决策矩阵与前端可视化拆分模式

**问题**:
- v0.2.53.28 已对齐"第三道门以 `business_writer_ready`(env AND impl)为准",但仅暴露合并字段,前端无法区分"env 已开但 Impl 未注入"vs"env 未开"(默认)。
- 路径 3 默认文案"BusinessWriter 未启用"会让用户在 env 已开 + Impl 未注入场景误以为是 env 未设。

**根因**:
- backend 决策矩阵 N 路径(8 路径)与 frontend 可视化 K 字段(K=3)不一致 — 一对一映射会让 frontend 复杂度爆炸。
- 文案边界化需求被低估:仅改 status code(403/501)不够,reason 文本也要对齐。

**解决**:
- **backend 决策矩阵定 N 路径(8 路径)+ frontend 收敛 K 字段(K=3)** — 沿用 v0.2.53.26 三门模式,backend 8 路径(本次新增 1 条 3.5-pre),frontend 3 字段独立展示 + 合并 ready tone 提示。
- **文案边界化** — 路径 3.5-pre 文案明确"env 已开且 Impl 未注入 + 需 DASHBOARD_REAL_DB=1 + session 成功 + Impl 构造成功",与路径 3 默认文案"BusinessWriter 未启用"形成区分。
- **`writer_impl_injected` None vs False**:
  - `None`:未追踪(默认 writer_env 不开路径,evaluate 不会读 ctx),不影响 status code
  - `False`:caller 显式传(沿 handler 透传 ctx),触发路径 3.5-pre 边界

**范本升级**:撞坑 #68 决策矩阵与可视化拆分模式 — 后续涉及 N 路径 → K 字段映射场景(例如 v0.2.53.30+ 多阶段业务 writer),沿用 backend N 路径 + frontend K 字段双轨收敛。
