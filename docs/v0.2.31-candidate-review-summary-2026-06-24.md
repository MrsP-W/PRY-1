## 2. 设计决策

### 2.1 三方案对比

| 方案 | 内容 | 优点 | 缺点 |
|------|------|------|------|
| A. 直接写汇总脚本 + 报告 | 一次性脚本 + 报告 + 1 测试 | 短平快 | 无回归保护,无 schema 引导 |
| **B. 双脚本(导出已硬化)+ 汇总脚本 + review_decision 闭环支持 + 14 tests**(本轮采用)| 沿 v0.2.30 范本(`.gitignore` + CLI 错误硬化 + 38 tests 级别),**支持可选 `review_decision` 列读取** + 三分类白名单 + 防 `## ##` 标题双 # 回归测试 | **向后兼容**(当前 1 行 CSV 无 decision 列也能跑)+ schema 引导(用户能照着标注) + 6 维度聚合 | 工程量稍大 |
| C. 全套:汇总 + 自动确认 stub + DB 回写 | 加 `--apply` 模式写回 DB | "自动化" | ❌ 越界,违背"不自动确认" |

### 2.2 决策:方案 B

**理由**:
- ✅ 沿 v0.2.30 范本(本地不入库 + CLI 错误硬化 + 38 tests 级别)
- ✅ 向后兼容 — 当前 `transaction-candidates.csv`(1 行无 decision 列)可正常出汇总
- ✅ 三分类语义清晰 — `same_transaction` / `separate_transactions` / `needs_investigation`,无歧义
- ✅ 不越界 — 不写回 DB,留待 v0.2.32+ 决策自动化路径
- ✅ 撞坑史沿用 — v0.2.18 §3 撞坑 #21(跨项目 Bash 显式 cd)+ #45(`uv run` 前缀)

## 5. 撞坑观察与教训

### 5.1 撞坑 #46(本轮新增)— 测试通过 ≠ 实测通过(markdown 标题双 #)

**现象**:13 tests 全绿,但实测跑出来 `## ## 2. 按 source 分布` 双 `##` 渲染 bug。

**根因**:
- `_format_count_table(title, ...)` 函数内部加 `## title`
- 调用方传入 `"## 2. 按 source 分布"` 已带 `##`
- 二者叠加 → `## ## 2. ...`
- 单元测试只 assert 值存在性(`alipay` / `wechat`),不检查 markdown 标题格式

**修法**:
- 调用方改传不带 `##` 的标题(`"2. 按 source 分布"`)
- 新增 `test_build_summary_report_no_double_hash_in_titles` 防回归:扫所有 `## ` 行,标题内部不能再含 `#`

**教训**:**测试覆盖度的边界 ≠ 实际渲染的边界**。值测试通过 ≠ 格式正确。沿 [[撞坑 #42]] — 严判边界要贴合业务语义,不要"看起来通过"。

### 5.2 撞坑 #47(本轮新增)— `_format_review_decision_table` 内部状态泄漏

**现象**:`total_decided/total_rows` 比例渲染成 `3/0`(应是 `3/3`)。

**根因**:函数用 `len(samples.get("_total", []))` 但 `_total` 字段从未被填充,默认 0。

**修法**:把 `total_rows` 提升为函数参数,从调用方显式传入。

**教训**:**隐式依赖 = 隐藏 bug**。函数签名应该显式表达所有外部依赖,不要靠"约定字段"。

### 5.3 撞坑 #48(本轮新增)— `_read_rows` 顺序错了

**现象**:`test_read_rows_unknown_suffix` 期望 `foo.xlsx` 抛 ValueError,但实际抛 FileNotFoundError。

**根因**:`_read_rows` 先检查 `exists()`,后 `_detect_format()`,文件不存在 → FileNotFoundError 优先,根本走不到 ValueError。

**修法**:交换顺序,先 `_detect_format()` 再 `exists()`。

**教训**:**校验顺序 = 错误语义优先级**。格式校验应该在"能不能打开文件"之前,否则用户传错扩展名 + 文件不存在会得到错误提示。
