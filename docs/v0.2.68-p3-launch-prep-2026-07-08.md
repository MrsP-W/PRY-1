## 🎯 P3 目标(用户授权范围)

**唯一目标**:在 v1.0 release 之前验证最小真实业务闭环:

- **链路完整**:用户输入 → LLM 草稿 → outbox 入库 → 1-click 审批 → SMTP 真发
- **5 重门控全绿**:撞坑 #76/#78/#79/#85/#86 边界 + 撞坑 #71 docs-only + 撞坑 #1 凭据安全
- **审计记录完整**:每个真实动作都有 audit_id 可追溯(LOG + UI 双通道)
- **失败回滚可恢复**:任何失败都不影响主库与 Keychain

**红线**:

- ❌ **不批量**:Notes 单次 ≤ 5 条 · SMTP 单次 = 1 封(撞坑 #78)
- ❌ **不默认收件人**:每次 SMTP 真发必须用户明确指定收件人(本项目红线 · 沿撞坑 #1)
- ❌ **不默真写**:Notes 真同步需用户逐条同意
- ❌ **不默认域白名单**:`SEND_REAL_NETWORK_RECIPIENT_DOMAINS` env 默认空 = 拒所有外发(Layer 3 默认最安全,撞坑 #85)
- ✅ **每次独立授权**:每个真实业务动作前必须用户明确"OK"

## 📋 P3 任务清单(3 类 · 独立授权)

### 任务 A — Notes 1-5 条真同步(独立授权)

**前置条件**(7 项全绿才启动):

| # | 条件 | 检查命令 | 期望输出 |
|---|------|---------|----------|
| 1 | Apple ID 已登录 macOS | `defaults read com.apple.icloud AppleID` | 有非空 Apple ID |
| 2 | Notes app 里有 ≥ 5 条候选笔记 | 肉眼确认 | 用户主动说明候选 ID 或时间窗 |
| 3 | `NOTE_PROVIDER=icloud` env 已设 | `echo $NOTE_PROVIDER` | `icloud` |
| 4 | NotesCipherImpl 不开(`ENABLE_NOTES_ENCRYPTION=0` / UNSET)| `env \| grep -i notes_encryption` | UNSET 或 0 |
| 5 | sync_notes CLI 可执行(4 退出码)| `uv run python -m my_ai_employee.scripts.sync_notes --help` | exit 0 + usage |
| 6 | 生产 DB 路径可访问 | `test -f "$HOME/Library/Application Support/my-ai-employee/data.db"` | exit 0 |
| 7 | 用户逐条明确指定哪 1-5 条 | 用户口头说明 | "同步 ID=X, Y, Z 这 3 条" |

**执行命令**(以 `--max-rows 5` 范本 · 但需用户指定具体 ID 降级到 `--note-id <id>` 范本):

```bash
# 推荐:用户先告诉具体 ID,逐个跑
uv run python -m my_ai_employee.scripts.sync_notes sync --note-id <ID1> --dry-run
# 确认无误后:
uv run python -m my_ai_employee.scripts.sync_notes sync --note-id <ID1>
# 重复上面 2 步 × N 次

# 不推荐:--max-rows 5(撞坑 #78 不鼓励批量)
```

**失败回滚**:

```bash
# Notes 真同步无"删除"概念(只插不删),所以失败不会污染主库
# 失败状态在 NoteStore.sync_status=FAILED,自然可见
# 用户可手动 `list_by_needs_confirm` 看 + 选 reset 路径
```

**审计位置**:

- `~/Library/Logs/MyAIEmployee/outbox.log` —— 如果是 note structurer 触发草稿
- DB 内 `notes.sync_status` 字段 —— 5 状态 NEW/STRUCTURED/PRIVATE_SKIP/FAILED/ARCHIVED
- `~/Library/Logs/MyAIEmployee/imap-sync.out.log` —— IMAP sync 阶段

### 任务 B — SMTP 单封真发(逐封命名收件人授权)

**前置条件**(10 项全绿才启动):

| # | 条件 | 检查命令 | 期望输出 |
|---|------|---------|----------|
| 1 | 用户**逐封命名收件人**(本次真发 1 封给谁)| 用户口头说明 | `<email>` 形式明确字符串 |
| 2 | Layer 3 env 已设 | `env \| grep SEND_REAL_NETWORK_RECIPIENT_DOMAINS` | 包含收件人域名 |
| 3 | IMAP/SMTP 凭据在 Keychain | `security find-generic-password -s my-ai-employee.imap.qq -a <email>` + `security find-generic-password -s my-ai-employee.smtp.qq -a <email>` | 均 exit 0 |
| 4 | outbox 生产库存在 | `test -f "$HOME/Library/Application Support/my-ai-employee/data.db"` | exit 0 |
| 5 | 草稿在 outbox status=approved | 用户说明草稿 ID | 已审批 |
| 6 | SEND_REAL_NETWORK=1 临时授权 | 用户口头 "OK 临时设 1" | 本次 shell 临时,不入 profile |
| 7 | SMTPProvider=qq 默认 | `env \| grep SMTP_PROVIDER` | `qq` |
| 8 | NO_REPLY 鉴权 16 位授权码 | Keychain | 已 Keychain 写入 |
| 9 | DAY3 已验证 1 封可发(基准)| docs/v0.2.55.5 报告 | `sent=1` |
| 10 | 撞坑 #85 LLM 草稿幻觉防御 3 层闭合 | docs/v0.2.66 + #85 报告 | Layer 1+2+3 OK |

**执行命令**:

```bash
# Layer 3 domain 白名单授权
SEND_REAL_NETWORK=1 SEND_REAL_NETWORK_RECIPIENT_DOMAINS=qq.com \
  uv run python -m my_ai_employee.scripts.process_inbox --execute --limit 1
# 用户确认:草稿 ID + 收件人 email + Subject 概要
```

**最小化红线**:

- ⚠️ **收件人必须用户口头逐封确认**(沿撞坑 #1 · 此红线不可绕过)
- ⚠️ **不配 outlook/gmail**(撞坑 #59)
- ⚠️ **不写 SEND_REAL_NETWORK_RECIPIENT_DOMAINS 到 shell profile**(每次临时授权)

**失败回滚**:

```bash
# 任何失败状态 outbox.status=pending_send(不变)
# 若已发:OutboxStore.update_status(id, status='cancelled') 不可能(已 sent=true)
# 若未发但进入中:OutboxDispatcher 异常隔离(撞坑 #71 P0 修复)
# 审计位置:audit_id 字符串 "audit:<uuid>" + approval_gate_audits 表 + outbox.log
```

**审计位置**:

- `approval_gate_audits` 表(DB 内)—— 每次真发留审计(actor + reason + timestamp + decision)
- Dashboard UI `/api/approval-gate/audits?limit=10` —— 可查最近 10 条审计
- `~/Library/Logs/MyAIEmployee/outbox.log` —— SMTP 实际交互日志
