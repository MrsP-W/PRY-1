## 5. 关键设计决策(7 条)

### 5.1 docs-only 启动 1 commit + 主代码 4 commits 拆 commit

**理由**:
- 沿 v0.2.2 范本(每个子阶段 2 commits:docs-only 启动 + feat+docs closure 收口)
- 启动文档先落地,清晰任务分解给 6/19-22 端午不休息期间
- 4 commits 每天 1 个,符合单日单 commit 节奏(D5.5.5 范本)

### 5.2 MicrosoftOAuth2 + GoogleOAuth2 独立文件(非合并)

**理由**:
- 沿 Phase 1 Protocol 抽象层,每个 provider 独立 `core/oauth2_microsoft.py` + `core/oauth2_google.py`
- 删一个 provider 不影响另一个(强一致)
- 后续可独立测试 mock(不复用 mock 类)

### 5.3 XOAUTH2 独立 `connectors/smtp/xoauth2.py` 模块

**理由**:
- 与现有 `connectors/smtp.py`(QQ 密码鉴权)解耦
- 未来 OutlookProvider/GmailProvider(候选 #1 工厂模式实化)直接复用 XOAUTH2 helper
- 单元测试独立,无 QQ SMTP 干扰

### 5.4 测试全部 mock,真实 OAuth flow 推迟

**理由**:
- OAuth 真实 flow 需用户 Microsoft/Google 账号 + App 注册 + 用户授权(无法 CI 自动化)
- 单元测试 mock MSAL/google-auth client 的 `acquire_token_*` 方法返回值
- 真实 spike 等用户手动操作(沿 B 类决策延后声明)

### 5.5 pyproject.toml 沿 `dependencies` 而非 `optional-dependencies.oauth`

**理由**:
- msal + google-auth 是 outlook/gmail SMTP 必需,不是可选
- v0.2.1 #6 Phase 1 launch 已有 `SERVICE_OAUTH_MICROSOFT` / `SERVICE_OAUTH_GOOGLE` 强契约
- 用户即使不用 outlook/gmail,装 msal/google-auth 也不影响(纯 Python,无系统依赖)

### 5.6 不实化 OutlookProvider / GmailProvider(候选 #1 工厂模式 docs-only 保持)

**理由**:
- Phase 2 仅完成 OAuth 抽象 + XOAUTH2 SMTP 鉴权 helper
- 工厂模式实化是独立候选(候选 #1 outlook/gmail SMTP provider · P0 · 6-8 commits)
- B 类决策延后声明:outlook/gmail SMTP provider 由用户单独决策启动

### 5.7 主代码 commits 全部跑 8/8 质量门(沿 v0.2.2 范本)

**理由**:
- 8 门:pytest + mypy src + mypy src+tests + ruff check + ruff format + alembic --sql + uv build + MD lint
- 单个 commit 失败立即回滚,不堆积
- 沿 v0.2.2 P0/#2/#3/#6/#7 严格单日单门 8 门全绿范本
