# 7/11–7/25 每日开发·检查·修复：无人值守运行 100% 计划

> **计划性质**：这是 15 天自动运行计划的完成度，不替代项目当前“可无人值守就绪度”基线（约 78%）。
> **目标**：到 7/25 形成可验证的 100% 计划闭环：持续运行、每日巡检、5% 节点检查/修复、证据留存。
> **红线**：不自动 push / 打 tag / 发邮件 / 写生产业务数据 / 启停 LaunchAgent；外部写入和服务启停仍需单独授权。

## 运行规则

1. 每天执行“开发 → 检查 → 安全修复 → 复测 → 留证”闭环；任一硬门失败，当日状态为阻塞，不累计进度。
2. 每完成 **5%**，必须执行一次项目检查与修复：`make check-snapshot`、`make ci`、只读 health 采样、失败归因、允许范围内的修复及复测。
3. 自动修复仅限无副作用项（格式化、确定性测试/快照一致性修复）；工作区非干净、需要业务判断、涉及凭据/网络/生产数据时只记录并阻塞，绝不强行修复。
4. 所有节点保存退出码、质量门摘要、health JSON 摘要和修复前后差异；证据不记录凭据、邮件正文或日志正文。

## 单个 5% 节点的自动执行顺序

```bash
# 1. 一致性与全量质量门
make check-snapshot
make ci

# 2. 只读运行时健康采样（不读取日志正文、不启停服务）
uv run python scripts/sample_launchd_health.py --format json
```

判定与动作：

| 结果 | 自动动作 | 进度 |
|---|---|---|
| 全部通过且 health=true | 记录摘要与时间戳 | 进入下一个 5% 节点 |
| 格式/快照等确定性问题 | 在干净工作区内执行最小修复，重新跑上述两项 | 复测通过才前进 |
| 测试失败、健康异常、工作区非干净或需业务判断 | 保存脱敏失败摘要，创建阻塞项；不自动改服务/外部系统 | 停留当前节点 |

## 验收口径（100%）

- 20/20 个 5% 节点都有检查记录；每个节点的修复与复测结果可追溯。
- 最近一轮 `make check-snapshot`、`make ci` 均通过。
- health 采样显示 menu-bar、Dashboard、127.0.0.1:8765 正常；计划型 job 的注册状态正常。
- 24h 与 72h 观察无未处理的重启、监听异常或新的 stderr 元数据异常。
- 全程未触发外部发送、生产业务写入、自动 push/tag 或未经授权的服务启停。

## 补跑规则

若计划建立时已过某个日期，不补写“已完成”。将该日期节点标记为 `待补跑`，按单个 5% 节点顺序执行；前一节点没有证据时，后续节点不得直接标记完成。
